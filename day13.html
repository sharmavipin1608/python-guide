Day 13 Phase 1 — Foundation

Type Hints + Dataclasses

Python is dynamically typed, but type hints give your IDE and static analysis tools everything they need to catch bugs before runtime. @dataclass then eliminates the boilerplate of writing __init__, __repr__, and __eq__ by hand.

Plan for the Day

What you'll cover

  • Why type hints exist and what enforces them (tooling, not Python itself)
  • Basic hints: str, int, float, bool, None
  • typing module: Optional, Union, list, dict, tuple, Any
  • Function signature hints and return types
  • @dataclass — auto-generated __init__, __repr__, __eq__
  • Dataclass fields: defaults, field(), frozen=True
  • ClassVar, InitVar, and __post_init__

Java parallels

  • Type hints → Java's explicit type declarations — but optional and not enforced at runtime
  • Optional[str] → Optional<String> — may be None
  • Union[int, str] → no direct equivalent (closest: generics or sealed types)
  • @dataclass → Lombok @Data or Java 16+ record
  • frozen=True dataclass → Java record (immutable by default)
  • field(default_factory=list) → Lombok's @Builder.Default

Type Hints

The basics — variable and function hints

Type hints are annotations — they're stored as metadata and ignored at runtime. Tools like mypy, pyright, and your IDE use them to catch type errors, suggest completions, and generate docs. Add them progressively; you don't have to annotate everything.

Java 
// Types are enforced at compile time
String name = "Alice";
int age = 30;
boolean active = true;

public String greet(String name, int times) {
    return name.repeat(times);
}

public Optional<String> findUser(int id) {
    return Optional.ofNullable(db.get(id));
}
Python 
# Variable hints — purely informational
name: str = "Alice"
age: int = 30
active: bool = True
score: float = 9.5

# Function hints — parameters and return type
def greet(name: str, times: int) -> str:
    return name * times

def find_user(user_id: int) -> str | None:
    # Python 3.10+: use X | Y instead of Union[X, Y]
    return users.get(user_id)

# Still works without hints — hints are never enforced
greet("Alice", 3)         # fine
greet(42, "oops")         # runs! (mypy would flag this)
Runtime behaviour: Python does NOT enforce type hints. def add(x: int, y: int) -> int: return x + y happily accepts strings. Type hints are a contract with your tools, not the interpreter.

Collection types

From Python 3.9+ you can use built-in types directly in hints (list[str], dict[str, int]). For older code you'll encounter typing.List, typing.Dict etc. — they're equivalent.

Python 
from typing import Any

# Python 3.9+ — use built-ins directly (preferred)
def process_names(names: list[str]) -> dict[str, int]:
    return {name: len(name) for name in names}

def flatten(matrix: list[list[int]]) -> list[int]:
    return [n for row in matrix for n in row]

# Tuples — fixed length, specify each element type
def get_coords() -> tuple[float, float]:
    return (51.5, -0.1)

# Tuples — variable length (homogeneous)
def get_scores() -> tuple[int, ...]:
    return (90, 85, 92, 88)

# Any — opt out of type checking for this value
def legacy_parse(data: Any) -> dict:
    return dict(data)

Optional, Union, and the new syntax

Python 
from typing import Optional, Union

# Optional[X] means the value can be X or None
def find(key: str) -> Optional[str]:   # classic style
    return cache.get(key)

def find_v2(key: str) -> str | None:   # Python 3.10+ (preferred)
    return cache.get(key)

# Union[X, Y] — one of several types
def parse_id(raw: Union[str, int]) -> int:   # classic
    return int(raw)

def parse_id_v2(raw: str | int) -> int:      # Python 3.10+
    return int(raw)

# Callable — type of a function
from typing import Callable

def apply(fn: Callable[[int, int], int], a: int, b: int) -> int:
    return fn(a, b)

apply(lambda x, y: x + y, 3, 4)  # 7

Type aliases and TypeVar

Python 
from typing import TypeVar, TypeAlias

# Type alias — give a complex type a readable name
UserId: TypeAlias = int
UserMap: TypeAlias = dict[UserId, str]

def get_users() -> UserMap:
    return {1: "Alice", 2: "Bob"}

# TypeVar — for generic functions (like Java generics)
T = TypeVar("T")

def first(items: list[T]) -> T | None:
    return items[0] if items else None

# mypy knows the return type matches the input element type
result: str | None = first(["a", "b", "c"])  # inferred as str | None

@dataclass

Eliminating boilerplate

@dataclass reads field annotations and auto-generates __init__, __repr__, and __eq__. You write field declarations; Python writes the methods. This is the Python equivalent of Lombok @Data or a Java record.

Java — record (Java 16+) 
// Java record — immutable, auto-generates
// constructor, accessors, equals, hashCode, toString
public record Point(int x, int y) {}

// Lombok @Data on a mutable class
@Data
public class User {
    private String name;
    private int age;
    private String email;
}
Python 
from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int
    # Auto-generated:
    # __init__(self, x: int, y: int)
    # __repr__ → Point(x=3, y=4)
    # __eq__   → compares field by field

p1 = Point(3, 4)
p2 = Point(3, 4)
p3 = Point(0, 0)

print(p1)         # Point(x=3, y=4)
print(p1 == p2)   # True  — value equality
print(p1 == p3)   # False
print(p1.x)       # 3 — direct attribute access

Default values and field()

Fields with defaults must come after fields without defaults (same rule as function parameters). For mutable defaults like lists, you must use field(default_factory=...) — never a bare [] as default.

Python 
from dataclasses import dataclass, field
from datetime import datetime

@dataclass
class Task:
    title: str                                  # required
    description: str = ""                       # optional with default
    done: bool = False
    tags: list[str] = field(default_factory=list)   # mutable default — MUST use field()
    created_at: datetime = field(default_factory=datetime.now)

    # DON'T do this — the list is shared across all instances!
    # bad_tags: list[str] = []   ← bug: mutable default


t1 = Task("Buy groceries")
t2 = Task("Write code", tags=["python", "backend"])

t1.tags.append("urgent")
print(t1.tags)   # ['urgent']
print(t2.tags)   # ['python', 'backend'] — separate list, correct!

frozen=True — immutable dataclasses

frozen=True makes instances immutable (like a Java record). Python raises FrozenInstanceError on attempted mutation, and also generates __hash__ so frozen dataclasses can be used in sets and as dict keys.

Python 
from dataclasses import dataclass

@dataclass(frozen=True)
class Color:
    r: int
    g: int
    b: int

    def to_hex(self) -> str:
        return f"#{self.r:02x}{self.g:02x}{self.b:02x}"


red = Color(255, 0, 0)
print(red)            # Color(r=255, g=0, b=0)
print(red.to_hex())   # #ff0000

# red.r = 128         # FrozenInstanceError!

# Hashable — can use in sets and as dict keys
palette = {red, Color(0, 255, 0), Color(0, 0, 255)}
theme = {Color(255, 0, 0): "danger", Color(0, 128, 0): "success"}

__post_init__, ClassVar, and field metadata

Python 
from dataclasses import dataclass, field, ClassVar
from typing import ClassVar as CV

@dataclass
class BankAccount:
    owner: str
    _balance: float = field(default=0.0, repr=False)  # hidden from repr

    # ClassVar — NOT a dataclass field, shared across instances
    interest_rate: ClassVar[float] = 0.05

    def __post_init__(self):
        # Called after __init__ — use for validation or derived fields
        if self._balance < 0:
            raise ValueError(f"Balance cannot be negative: {self._balance}")
        self.account_id = f"ACC-{id(self):x}"   # derived field

    def deposit(self, amount: float) -> None:
        self._balance += amount

    @property
    def balance(self) -> float:
        return self._balance


acc = BankAccount("Alice", 1000.0)
print(acc)               # BankAccount(owner='Alice') — _balance hidden (repr=False)
print(acc.balance)       # 1000.0
print(acc.account_id)    # ACC-... (derived)
print(BankAccount.interest_rate)  # 0.05

dataclass vs NamedTuple vs TypedDict

Quick decision guide:
Use @dataclass when you need a mutable object with methods and validation.
Use @dataclass(frozen=True) when you need an immutable value object (like a Java record).
Use NamedTuple when you want tuple unpacking + named fields + immutability (no methods).
Use TypedDict when working with JSON/dicts and you want type hints on keys without a class.

Wrap-up

  • Type hints are a contract with tools, not the runtime. Python never enforces them — but mypy, pyright, and your IDE use them to catch bugs, power autocomplete, and generate documentation. Add them from the start on new code.
  • Use X | None (Python 3.10+) instead of Optional[X]. Use X | Y instead of Union[X, Y]. Use built-in list[str] instead of typing.List[str]. The old forms still work but are verbose.
  • @dataclass generates __init__, __repr__, and __eq__ from your field annotations — it's Lombok @Data or a Java Record, built into the standard library since Python 3.7.
  • Never use a mutable object as a bare default (tags: list = []). That list is shared across all instances. Always use field(default_factory=list) — Python will flag this as an error if you forget.
  • frozen=True gives you immutability and hashability — the Python equivalent of a Java record. Use it for value objects that should never change after creation (coordinates, money amounts, config entries).

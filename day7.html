Mini Project: CLI Task Manager

Build a command-line task manager that persists data to JSON. Apply everything you've learned: functions, data structures, file I/O, error handling, and modules.

Project Overview

What to build

A CLI app that lets you add, list, complete, and delete tasks. Tasks are persisted to a JSON file between runs.

Inputs

Command-line interaction (menu-driven) — user selects actions from a numbered menu.

Outputs

tasks.json file for persistence, formatted task list in the terminal.

Skills covered

Functions (Day 3), data structures (Day 4), file I/O + JSON (Day 5), error handling + modules (Day 6).

Stretch goal

Add due dates with datetime and filter tasks by status (pending / done).

Architecture and Design

Before writing code, plan the structure. As a Java developer, you'd normally create a class hierarchy. For this project, we'll keep it functional (no classes yet — that's Day 8). Think of it as writing a Spring Boot service where every @Service method is a standalone function.

File structure

Project layout 
task_manager/
├── task_manager.py   ← main entry point (menu loop)
├── storage.py        ← load/save tasks to JSON
└── tasks.json        ← data file (created at runtime)

Task representation

Each task is a dict with a unique ID, a title, a status, and a creation timestamp. In Java you'd make a Task class — here a dict suffices. (On Day 14 we'll refactor this to use a @dataclass.)

Python — task shape 
# Each task is a dict:
{
    "id": 1,
    "title": "Buy groceries",
    "done": False,
    "created_at": "2024-03-15T10:30:00"
}

Separation of concerns

Design principle: Keep I/O (reading input, printing output, file access) separate from logic (finding tasks, toggling status). This mirrors the service/repository pattern you know from Spring. storage.py handles persistence; task_manager.py handles the user-facing logic and menu.

Step 1: The Storage Module

Start with the persistence layer — loading and saving tasks to a JSON file. This module has no user interaction; it only reads and writes data.

Python — storage.py 
"""
storage.py — Load and save tasks to a JSON file.
"""
import json
from pathlib import Path

DEFAULT_FILE = Path("tasks.json")


def load_tasks(path=DEFAULT_FILE):
    """Load tasks from a JSON file. Return empty list if file doesn't exist."""
    try:
        with open(path) as f:
            return json.load(f)
    except FileNotFoundError:
        return []
    except json.JSONDecodeError:
        print(f"Warning: {path} contains invalid JSON. Starting fresh.")
        return []


def save_tasks(tasks, path=DEFAULT_FILE):
    """Save the task list to a JSON file."""
    with open(path, "w") as f:
        json.dump(tasks, f, indent=2)


def get_next_id(tasks):
    """Return the next available task ID."""
    if not tasks:
        return 1
    return max(task["id"] for task in tasks) + 1
Java comparison: This module is like a Spring @Repository — it abstracts the data layer. load_tasks() ≈ findAll(), save_tasks() ≈ the underlying JPA flush, get_next_id() ≈ an auto-increment sequence.

Step 2: The Main Application

Now build the menu system and task operations. Each menu option maps to a function — like controller methods routing to service calls.

Python — task_manager.py 
"""
task_manager.py — CLI Task Manager.
Run with: python task_manager.py
"""
from datetime import datetime
from storage import load_tasks, save_tasks, get_next_id


def display_menu():
    """Print the main menu."""
    print("\n╔══════════════════════════╗")
    print("║    📋 Task Manager       ║")
    print("╠══════════════════════════╣")
    print("║  1. Add task             ║")
    print("║  2. List tasks           ║")
    print("║  3. Complete task        ║")
    print("║  4. Delete task          ║")
    print("║  5. Quit                 ║")
    print("╚══════════════════════════╝")


def add_task(tasks):
    """Prompt user for a task title and add it to the list."""
    title = input("Task title: ").strip()
    if not title:
        print("Error: Title cannot be empty.")
        return

    task = {
        "id": get_next_id(tasks),
        "title": title,
        "done": False,
        "created_at": datetime.now().isoformat(),
    }
    tasks.append(task)
    save_tasks(tasks)
    print(f"Added: '{title}' (ID: {task['id']})")


def list_tasks(tasks):
    """Display all tasks with their status."""
    if not tasks:
        print("No tasks yet. Add one with option 1.")
        return

    print(f"\n{'ID':<5} {'Status':<10} {'Title'}")
    print("-" * 40)
    for task in tasks:
        status = "✓ Done" if task["done"] else "○ Pending"
        print(f"{task['id']:<5} {status:<10} {task['title']}")
    print(f"\nTotal: {len(tasks)} task(s)")


def complete_task(tasks):
    """Mark a task as complete by ID."""
    list_tasks(tasks)
    if not tasks:
        return

    try:
        task_id = int(input("\nTask ID to complete: "))
    except ValueError:
        print("Error: Please enter a number.")
        return

    for task in tasks:
        if task["id"] == task_id:
            if task["done"]:
                print(f"Task '{task['title']}' is already done.")
            else:
                task["done"] = True
                save_tasks(tasks)
                print(f"Completed: '{task['title']}'")
            return

    print(f"Error: No task with ID {task_id}.")


def delete_task(tasks):
    """Delete a task by ID."""
    list_tasks(tasks)
    if not tasks:
        return

    try:
        task_id = int(input("\nTask ID to delete: "))
    except ValueError:
        print("Error: Please enter a number.")
        return

    for i, task in enumerate(tasks):
        if task["id"] == task_id:
            removed = tasks.pop(i)
            save_tasks(tasks)
            print(f"Deleted: '{removed['title']}'")
            return

    print(f"Error: No task with ID {task_id}.")


def main():
    """Main loop — load tasks, show menu, dispatch actions."""
    tasks = load_tasks()
    print("Loaded", len(tasks), "task(s).")

    actions = {
        "1": add_task,
        "2": list_tasks,
        "3": complete_task,
        "4": delete_task,
    }

    while True:
        display_menu()
        choice = input("Choose (1-5): ").strip()

        if choice == "5":
            print("Goodbye!")
            break

        action = actions.get(choice)
        if action:
            action(tasks)
        else:
            print("Invalid choice. Please enter 1-5.")


if __name__ == "__main__":
    main()

Key Patterns Used

This project applies specific patterns from each day. Here's where they show up:

Dispatch dict (Day 3 — functions as first-class objects)

Python 
# Instead of a long if/elif chain:
if choice == "1":
    add_task(tasks)
elif choice == "2":
    list_tasks(tasks)
# ... etc

# We use a dispatch dict — cleaner and extensible:
actions = {
    "1": add_task,
    "2": list_tasks,
    "3": complete_task,
    "4": delete_task,
}
action = actions.get(choice)
if action:
    action(tasks)

try/except for input validation (Day 6)

Python 
# Guard against non-numeric input
try:
    task_id = int(input("Task ID: "))
except ValueError:
    print("Error: Please enter a number.")
    return

# Guard against missing file (in storage.py)
try:
    with open(path) as f:
        return json.load(f)
except FileNotFoundError:
    return []

enumerate for indexed iteration (Day 4)

Python 
# enumerate gives us (index, item) — needed for pop(i)
for i, task in enumerate(tasks):
    if task["id"] == task_id:
        removed = tasks.pop(i)    # remove by index
        break

__name__ == '__main__' guard (Day 6)

Python 
# Allows importing task_manager functions without running the menu loop
if __name__ == "__main__":
    main()

# Now you could do: from task_manager import add_task
# without triggering the interactive menu

Wrap-up

  • You built a working CLI app with persistent storage in about 100 lines of Python. The equivalent Java version would be significantly more boilerplate (POJOs, builders, try-with-resources, Jackson config).
  • The dispatch dict pattern (actions = {"1": func}) replaces verbose if/elif chains. Functions as first-class objects make this natural in Python.
  • Separating storage from application logic (even without classes) keeps the code clean and testable. storage.py could be swapped for a database layer later.
  • Defensive coding with try/except around user input prevents crashes. The app gracefully handles missing files, invalid input, and edge cases.
  • The __name__ guard lets your module be both a standalone script and an importable library — a pattern you'll use in every Python project.

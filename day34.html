Environment + Config Patterns

Twelve-factor apps keep config out of code. Python has excellent tools for this — python-dotenv for loading .env files and Pydantic BaseSettings for typed, validated config that maps cleanly to Spring's @ConfigurationProperties.

1. python-dotenv

python-dotenv reads a .env file and loads its key-value pairs into os.environ. It's the simplest way to manage local config without hardcoding values.

$ pip install python-dotenv
.env 
DATABASE_URL=postgresql://user:pass@localhost:5432/mydb
SECRET_KEY=dev-secret-key-change-in-prod
DEBUG=true
LOG_LEVEL=debug
ALLOWED_HOSTS=localhost,127.0.0.1
Python 
from dotenv import load_dotenv
import os

# Load .env into os.environ (call this as early as possible)
load_dotenv()

db_url = os.environ["DATABASE_URL"]       # raises KeyError if missing
secret = os.getenv("SECRET_KEY", "fallback")  # safe, with default

# Override: already-set env vars take precedence by default
# To force .env values even when env var exists:
load_dotenv(override=True)

# Load a specific file
load_dotenv(".env.local")
Always add .env to .gitignore. Commit a .env.example with placeholder values instead — this documents what config is needed without exposing real secrets.

2. Pydantic BaseSettings

BaseSettings is the proper way to manage config in production Python apps. It reads from environment variables (and optionally .env), validates types, and gives you IDE autocomplete — just like @ConfigurationProperties in Spring.

$ pip install pydantic-settings
Java — @ConfigurationProperties 
@ConfigurationProperties(prefix = "app")
@Validated
public class AppConfig {
    @NotBlank
    private String databaseUrl;

    @NotBlank
    private String secretKey;

    private boolean debug = false;

    private int port = 8000;

    @Min(1) @Max(100)
    private int maxConnections = 10;

    // getters/setters ...
}
Python — BaseSettings 
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import field_validator, PostgresDsn

class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        env_file=".env",          # load from .env file
        env_file_encoding="utf-8",
        case_sensitive=False,     # DATABASE_URL == database_url
    )

    database_url: PostgresDsn    # validates it's a real Postgres URL
    secret_key: str
    debug: bool = False
    port: int = 8000
    max_connections: int = 10

    @field_validator("max_connections")
    @classmethod
    def check_connections(cls, v: int) -> int:
        if not 1 <= v <= 100:
            raise ValueError("must be 1-100")
        return v

settings = Settings()            # raises if required fields missing

Using Settings in FastAPI

Python 
from functools import lru_cache
from fastapi import FastAPI, Depends
from config import Settings

app = FastAPI()

@lru_cache          # instantiate once, reuse (like a singleton bean)
def get_settings() -> Settings:
    return Settings()

@app.get("/info")
def info(settings: Settings = Depends(get_settings)):
    return {
        "debug": settings.debug,
        "port": settings.port,
        # never return secret_key!
    }
@lru_cache on get_settings means Settings() is only constructed once per process — environment variables are read once at startup, not on every request. This is the Python equivalent of a Spring singleton bean.

3. Environment-Specific Config

Different environments (dev, staging, prod) need different values. Python doesn't have Spring's built-in profile system, but it's straightforward to replicate with an ENVIRONMENT variable and layered settings.

Pattern 1 — Multiple .env Files

Python 
import os
from pydantic_settings import BaseSettings, SettingsConfigDict

ENV = os.getenv("ENVIRONMENT", "development")

class Settings(BaseSettings):
    model_config = SettingsConfigDict(
        # Load base, then environment-specific override
        env_file=(".env", f".env.{ENV}"),
        extra="ignore",
    )

    database_url: str
    secret_key: str
    debug: bool = False
    log_level: str = "info"

# .env           — base defaults
# .env.development  — dev overrides (gitignored)
# .env.production   — prod overrides (never committed, injected by CI/CD)

Pattern 2 — Settings Subclasses

Python 
from pydantic_settings import BaseSettings

class BaseConfig(BaseSettings):
    app_name: str = "TaskManager"
    api_prefix: str = "/api/v1"
    secret_key: str
    database_url: str

class DevelopmentConfig(BaseConfig):
    debug: bool = True
    log_level: str = "debug"
    database_url: str = "sqlite:///./dev.db"

class ProductionConfig(BaseConfig):
    debug: bool = False
    log_level: str = "warning"
    # database_url must come from environment

class TestingConfig(BaseConfig):
    debug: bool = True
    database_url: str = "sqlite:///:memory:"
    secret_key: str = "test-key"

def get_config() -> BaseConfig:
    env = os.getenv("ENVIRONMENT", "development")
    configs = {
        "development": DevelopmentConfig,
        "production": ProductionConfig,
        "testing": TestingConfig,
    }
    return configs[env]()

settings = get_config()

4. Secrets Management Patterns

For production, secrets should not live in files at all — they should come from a secrets manager. Here's how Python apps integrate with common options.

AWS Secrets Manager

Python 
import boto3
import json
from functools import lru_cache

@lru_cache
def get_secret(secret_name: str) -> dict:
    """Fetch a secret from AWS Secrets Manager. Cached after first call."""
    client = boto3.client("secretsmanager", region_name="us-east-1")
    response = client.get_secret_value(SecretId=secret_name)
    return json.loads(response["SecretString"])

# Usage
db_secret = get_secret("myapp/prod/database")
DATABASE_URL = (
    f"postgresql://{db_secret['username']}:{db_secret['password']}"
    f"@{db_secret['host']}:{db_secret['port']}/{db_secret['dbname']}"
)

BaseSettings + Secrets Manager Integration

Python 
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import model_validator

class Settings(BaseSettings):
    # In prod: these come from env vars set by the deployment platform
    # In dev: loaded from .env file
    database_url: str = ""
    secret_key: str = ""

    # AWS secret name (non-sensitive, can be in env)
    aws_secret_name: str = ""

    model_config = SettingsConfigDict(env_file=".env")

    @model_validator(mode="after")
    def load_from_secrets_manager(self) -> "Settings":
        if self.aws_secret_name:
            secrets = get_secret(self.aws_secret_name)
            self.database_url = secrets["database_url"]
            self.secret_key = secrets["secret_key"]
        return self
Secret rotation: If you cache secrets with @lru_cache, your app won't pick up rotated secrets without a restart. For high-security apps, add a TTL-based cache or reload from Secrets Manager periodically.

5. 12-Factor App Config Principles

The 12-factor app methodology defines best practices for building deployable services. Factor III (Config) is the most directly relevant here.

The Rules

Python — 12-factor checklist 
# ✅ DO: Read config from environment
DATABASE_URL = os.environ["DATABASE_URL"]

# ❌ DON'T: Hardcode env-specific values in code
DATABASE_URL = "postgresql://user:pass@localhost/mydb"

# ✅ DO: Fail fast if required config is missing
class Settings(BaseSettings):
    database_url: str          # required — raises at startup if absent
    secret_key: str            # required

# ❌ DON'T: Fall back silently to insecure defaults in prod
SECRET_KEY = os.getenv("SECRET_KEY", "changeme")  # dangerous

# ✅ DO: Keep dev/staging/prod config identical in structure
# (only values differ, not the keys or format)

# ✅ DO: Never commit secrets to version control
# Use .env.example with placeholder values

# ✅ DO: Treat config as environment-agnostic code
# The same Docker image runs in dev and prod — only env vars change

Practical Config File Layout

project layout 
myapp/
  .env             # local dev values (gitignored)
  .env.example     # template with placeholders (committed)
  .env.test        # values for pytest runs (committed if no secrets)
  config.py        # BaseSettings class
  main.py

# .gitignore must include:
.env
.env.*
!.env.example    # keep the example

Wrap-up

  • python-dotenv loads .env into os.environ. Use it for local dev. In Docker/prod, set env vars directly — no .env file needed.
  • Pydantic BaseSettings is the typed, validated approach. It reads from environment variables, validates types, and raises at startup if required config is missing — fail-fast beats silent failures.
  • @lru_cache on your settings factory makes it a singleton — constructed once at startup, reused everywhere via FastAPI's Depends().
  • Commit .env.example, never .env. The example documents what config is needed; the real file stays local and gitignored.
  • In production, secrets come from a secrets manager (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager) — not from files on disk.

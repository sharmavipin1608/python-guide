What Are Dunder Methods?

Dunder methods (short for "double underscore") are how Python implements operator overloading and built-in function hooks. When you write len(x), Python calls x.__len__(). When you write a == b, Python calls a.__eq__(b). You implement these methods to make your classes behave like built-ins.

__str__ and __repr__

__str__ vs __repr__

__str__ is for human-readable output — called by print() and str(). __repr__ is for unambiguous developer output — shown in the REPL, logs, and error messages. The rule: __repr__ should ideally be a string you could paste back into Python to recreate the object.

Java 
public class Point {
    private int x, y;

    public Point(int x, int y) {
        this.x = x; this.y = y;
    }

    @Override
    public String toString() {
        return "Point(" + x + ", " + y + ")";
    }
    // No __repr__ equivalent in Java
}
Python 
class Point:
    def __init__(self, x: int, y: int):
        self.x = x
        self.y = y

    def __str__(self) -> str:
        # Human-readable — for print()
        return f"({self.x}, {self.y})"

    def __repr__(self) -> str:
        # Unambiguous — for REPL, logs, debugging
        # Aim: eval(repr(obj)) == obj
        return f"Point({self.x!r}, {self.y!r})"


p = Point(3, 4)
print(p)        # (3, 4)        — uses __str__
print(repr(p))  # Point(3, 4)  — uses __repr__

# In a list, __repr__ is used for each element
points = [Point(1, 2), Point(3, 4)]
print(points)   # [Point(1, 2), Point(3, 4)]
Tip: If you only implement one, implement __repr__. Python falls back to __repr__ when __str__ is missing, so you get both contexts covered.

__eq__ and __hash__

Value equality vs identity

By default, Python classes use identity for == (same as is — same object in memory). To get value equality like Java's equals(), implement __eq__. Whenever you implement __eq__, you should also implement __hash__ so objects can be used in sets and as dict keys.

Java 
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Point)) return false;
    Point p = (Point) o;
    return x == p.x && y == p.y;
}

@Override
public int hashCode() {
    return Objects.hash(x, y);
}
Python 
class Point:
    def __init__(self, x: int, y: int):
        self.x = x
        self.y = y

    def __eq__(self, other) -> bool:
        if not isinstance(other, Point):
            return NotImplemented   # not False — lets Python try other.__eq__
        return self.x == other.x and self.y == other.y

    def __hash__(self) -> int:
        # Must be consistent with __eq__
        # Objects that are == must have the same hash
        return hash((self.x, self.y))

    def __repr__(self):
        return f"Point({self.x}, {self.y})"


p1 = Point(3, 4)
p2 = Point(3, 4)
p3 = Point(0, 0)

print(p1 == p2)  # True  — value equality
print(p1 is p2)  # False — different objects
print(p1 == p3)  # False

# Can use in sets and dicts because __hash__ is implemented
point_set = {p1, p2, p3}
print(len(point_set))  # 2 — p1 and p2 are "equal"

lookup = {p1: "origin-ish", p3: "origin"}
print(lookup[p2])  # "origin-ish" — p2 hashes/equals to p1
Return NotImplemented, not False when the type is wrong in __eq__. This signals Python to try the reflected operation on the other object — important for symmetry with custom types.

Container Dunders

__len__, __contains__, __getitem__

Implement these three and your class works with len(), in, and [] indexing. This is how Python's built-in list, dict, and str expose their behaviour — you can do the same for any custom collection.

Python 
class Playlist:
    def __init__(self, name: str):
        self.name = name
        self._songs = []

    def add(self, song: str) -> None:
        self._songs.append(song)

    def __len__(self) -> int:
        # Enables: len(playlist)
        return len(self._songs)

    def __contains__(self, song: str) -> bool:
        # Enables: "song" in playlist
        return song in self._songs

    def __getitem__(self, index: int) -> str:
        # Enables: playlist[0], playlist[-1], playlist[1:3]
        return self._songs[index]

    def __bool__(self) -> bool:
        # Enables: if playlist:
        # Without this, __len__ is used (0 == False, non-zero == True)
        return len(self._songs) > 0

    def __repr__(self):
        return f"Playlist({self.name!r}, {len(self)} songs)"


p = Playlist("Morning Mix")
p.add("Song A")
p.add("Song B")
p.add("Song C")

print(len(p))            # 3
print("Song B" in p)     # True
print("Song Z" in p)     # False
print(p[0])              # Song A
print(p[-1])             # Song C
print(p[1:3])            # ['Song B', 'Song C']

if p:
    print(f"{p} is not empty")
Bonus: Once you implement __getitem__ with integer indices, Python automatically makes your object iterable — you can use it in a for loop without implementing __iter__. (Day 10 covers iterators properly.)

@property — Better Than Getters/Setters

The Java getter/setter problem

Java forces you to write getX()/setX() from the start "just in case" you need validation later — or refactor the entire codebase if you start with a public field. Python's @property solves this: start with a plain attribute, add computed/validated access later with zero change to calling code.

Java 
public class Circle {
    private double radius;

    public Circle(double radius) {
        if (radius < 0) throw new IllegalArgumentException();
        this.radius = radius;
    }

    public double getRadius() { return radius; }

    public void setRadius(double r) {
        if (r < 0) throw new IllegalArgumentException();
        this.radius = r;
    }

    public double getArea() {
        return Math.PI * radius * radius;
    }
}
// Usage: c.getRadius(), c.setRadius(5), c.getArea()
Python 
import math

class Circle:
    def __init__(self, radius: float):
        self.radius = radius   # calls the setter below

    @property
    def radius(self) -> float:
        # Getter — called on attribute READ: c.radius
        return self._radius

    @radius.setter
    def radius(self, value: float) -> None:
        # Setter — called on attribute WRITE: c.radius = 5
        if value < 0:
            raise ValueError(f"Radius must be non-negative, got {value}")
        self._radius = value

    @property
    def area(self) -> float:
        # Read-only computed property — no setter needed
        return math.pi * self._radius ** 2

    @property
    def diameter(self) -> float:
        return self._radius * 2

    def __repr__(self):
        return f"Circle(radius={self._radius})"


c = Circle(5)
print(c.radius)    # 5       — looks like attribute access, calls getter
print(c.area)      # 78.53   — computed on the fly, no ()
print(c.diameter)  # 10

c.radius = 10      # calls setter — validated
print(c.area)      # 314.15

# c.radius = -1    # raises ValueError
# c.area = 100     # AttributeError: can't set attribute (no setter defined)
Zero refactor rule: If you start with self.radius = radius (plain attribute) and later need validation, you just add @property and rename the backing field to self._radius. All callers using c.radius keep working — no API change.

@property.deleter

You can also define a @property.deleter to handle del obj.attr. Rarely needed, but useful for resources and caches.

Python 
class CachedReport:
    def __init__(self):
        self._cache = None

    @property
    def data(self):
        if self._cache is None:
            print("Computing expensive data...")
            self._cache = [1, 2, 3]   # expensive computation
        return self._cache

    @data.deleter
    def data(self):
        # Clears the cache — called on: del report.data
        print("Cache cleared")
        self._cache = None


r = CachedReport()
print(r.data)  # Computing expensive data... [1, 2, 3]
print(r.data)  # [1, 2, 3]  — from cache, no recompute
del r.data     # Cache cleared
print(r.data)  # Computing expensive data... [1, 2, 3]

Putting it all together — a Money class

A realistic example combining __init__, __repr__, __str__, __eq__, __hash__, __add__, and @property:

Python 
from decimal import Decimal

class Money:
    def __init__(self, amount: str | Decimal, currency: str = "USD"):
        self._amount = Decimal(str(amount))
        self._currency = currency.upper()

    @property
    def amount(self) -> Decimal:
        return self._amount

    @property
    def currency(self) -> str:
        return self._currency

    def __str__(self) -> str:
        return f"{self._currency} {self._amount:.2f}"

    def __repr__(self) -> str:
        return f"Money({self._amount!r}, {self._currency!r})"

    def __eq__(self, other) -> bool:
        if not isinstance(other, Money):
            return NotImplemented
        return self._amount == other._amount and self._currency == other._currency

    def __hash__(self) -> int:
        return hash((self._amount, self._currency))

    def __add__(self, other: "Money") -> "Money":
        if self._currency != other._currency:
            raise ValueError(f"Cannot add {self._currency} and {other._currency}")
        return Money(self._amount + other._amount, self._currency)

    def __lt__(self, other: "Money") -> bool:
        if self._currency != other._currency:
            raise ValueError("Cannot compare different currencies")
        return self._amount < other._amount


price = Money("9.99")
tax   = Money("0.80")
total = price + tax

print(total)           # USD 10.79
print(repr(total))     # Money(Decimal('10.79'), 'USD')
print(price < total)   # True
print(sorted([total, price, tax]))  # sorted uses __lt__

Quick Reference Card

Human display
def __str__(self): return "..."

Called by print() and str(). Like Java's toString().

Debug display
def __repr__(self): return f"Class({self.x!r})"

REPL, logs, containers. If only one, implement this — __str__ falls back to it.

Value equality
def __eq__(self, other): return self.x == other.x

Enables ==. Return NotImplemented (not False) for wrong types.

Hashable
def __hash__(self): return hash((self.x, self.y))

Required if __eq__ is implemented. Allows use in set and as dict key.

len() support
def __len__(self): return len(self._items)

Enables len(obj) and makes empty object falsy by default.

in operator
def __contains__(self, item): return item in self._data

Enables x in obj.

Index access
def __getitem__(self, key): return self._data[key]

Enables obj[0], obj["key"], slices. Also makes object iterable.

@property getter
@property / def name(self): return self._name

Access as obj.name — no parentheses needed.

@property setter
@name.setter / def name(self, value): self._name = value

Called on obj.name = "x". Validate here.

Wrap-up

  • Dunder methods are how Python maps syntax (==, len(), in, [], print()) to your class. Implement them to make your objects behave like built-ins, not like foreign objects that need special method calls.
  • Always implement __repr__ — it serves as fallback for __str__ and is invaluable in logs and debugging. Make it reconstructable: eval(repr(obj)) == obj when practical.
  • If you implement __eq__, you must also implement __hash__. Python automatically sets __hash__ = None (making objects unhashable) when you define __eq__ without __hash__.
  • @property replaces Java's getter/setter boilerplate. Start with a plain attribute; add @property later for validation or computation — with zero change to calling code.
  • Read-only computed properties (like area) replace methods like getArea(). They're accessed without (), making the API cleaner — they look like data, not behaviour.

Day 08 Phase 1 — Foundation

OOP: Classes + Inheritance

Python's object model is familiar but different. No access modifiers, no interfaces, no compile-time type enforcement — yet the OOP story is complete and expressive.

Plan for the Day

What you'll cover

  • Defining classes with class and the __init__ constructor
  • self — Python's explicit this
  • Instance variables vs class variables
  • Instance methods, and how they differ from Java instance methods
  • Inheritance with super() and method overriding
  • No access modifiers — _protected and __mangled conventions
  • Duck typing — why interfaces aren't needed (yet)

Java parallels

  • class Foo → same syntax, no public keyword needed
  • public Foo(...) {} → def __init__(self, ...):
  • this.x = x → self.x = x
  • extends → class Child(Parent):
  • super().method() → same, but super() needs no args in Python 3
  • private → __name (name-mangled); protected → _name (convention only)

Defining Classes

Basic class structure

A Python class looks similar to Java but is leaner — no field declarations outside __init__, no access modifiers, and self must always be the first parameter of instance methods.

Java 
public class BankAccount {
    private String owner;
    private double balance;

    public BankAccount(String owner, double balance) {
        this.owner = owner;
        this.balance = balance;
    }

    public void deposit(double amount) {
        this.balance += amount;
    }

    public double getBalance() {
        return this.balance;
    }
}
Python 
class BankAccount:
    def __init__(self, owner: str, balance: float = 0.0):
        self.owner = owner      # instance variable
        self.balance = balance  # instance variable

    def deposit(self, amount: float) -> None:
        self.balance += amount

    def get_balance(self) -> float:
        return self.balance


acc = BankAccount("Alice", 1000.0)
acc.deposit(500)
print(acc.get_balance())  # 1500.0
self is not magic — it's just a convention. Python passes the instance as the first argument to every instance method. You could name it anything, but self is the universal convention. Java hides this with the implicit this.

Instance variables vs class variables

Instance variables are set on self — each object has its own copy. Class variables are defined in the class body and shared across all instances, like Java's static fields.

Python 
class Employee:
    # Class variable — shared by ALL instances (like Java static field)
    company = "Acme Corp"
    headcount = 0

    def __init__(self, name: str, salary: float):
        # Instance variables — unique per object
        self.name = name
        self.salary = salary
        Employee.headcount += 1  # access class var via class name

    def describe(self) -> str:
        # Both accessible on self, but class var is on the class
        return f"{self.name} at {Employee.company}"


e1 = Employee("Alice", 90_000)
e2 = Employee("Bob", 85_000)

print(Employee.headcount)   # 2
print(e1.company)           # "Acme Corp" — found via class
Employee.company = "MegaCorp"
print(e2.company)           # "MegaCorp" — all instances see the change
Gotcha: If you do e1.company = "Other" you create an instance variable that shadows the class variable on e1 only. The class variable is unchanged. This surprises Java developers.

Instance Methods

Regular instance methods

Any method defined with self as the first parameter is an instance method. Python has no separate syntax for void — methods return None by default.

Python 
class Stack:
    def __init__(self):
        self._items = []   # _prefix = "please treat as private"

    def push(self, item) -> None:
        self._items.append(item)

    def pop(self):
        if self.is_empty():
            raise IndexError("pop from empty stack")
        return self._items.pop()

    def peek(self):
        if self.is_empty():
            raise IndexError("peek at empty stack")
        return self._items[-1]

    def is_empty(self) -> bool:
        return len(self._items) == 0

    def size(self) -> int:
        return len(self._items)


s = Stack()
s.push(10)
s.push(20)
s.push(30)
print(s.peek())  # 30
print(s.pop())   # 30
print(s.size())  # 2

Static and class methods

Python has two alternatives to regular instance methods: @staticmethod (like Java static) and @classmethod (receives the class, not the instance — useful for factory methods).

Python 
class Temperature:
    def __init__(self, celsius: float):
        self.celsius = celsius

    # @staticmethod: no self, no cls — pure function scoped to class
    @staticmethod
    def celsius_to_fahrenheit(c: float) -> float:
        return c * 9 / 5 + 32

    # @classmethod: receives the class as first arg (cls), not instance
    # Great for alternative constructors (factory methods)
    @classmethod
    def from_fahrenheit(cls, f: float) -> "Temperature":
        return cls((f - 32) * 5 / 9)

    def __repr__(self):
        return f"Temperature({self.celsius:.1f}°C)"


t1 = Temperature(100)
print(Temperature.celsius_to_fahrenheit(100))  # 212.0

t2 = Temperature.from_fahrenheit(212)          # factory constructor
print(t2)                                       # Temperature(100.0°C)
Java parallel: @classmethod with an alternative constructor pattern replaces Java's multiple constructor overloads or static Foo of(...) factory methods.

Inheritance

Single inheritance with super()

Python inheritance uses parentheses: class Child(Parent). super() in Python 3 needs no arguments — it already knows which class and instance to use.

Java 
public class Animal {
    protected String name;
    public Animal(String name) {
        this.name = name;
    }
    public String speak() {
        return "...";
    }
}

public class Dog extends Animal {
    private String breed;
    public Dog(String name, String breed) {
        super(name);
        this.breed = breed;
    }
    @Override
    public String speak() {
        return "Woof!";
    }
}
Python 
class Animal:
    def __init__(self, name: str):
        self.name = name

    def speak(self) -> str:
        return "..."

    def __repr__(self):
        return f"{type(self).__name__}({self.name!r})"


class Dog(Animal):
    def __init__(self, name: str, breed: str):
        super().__init__(name)   # no args needed in Python 3
        self.breed = breed

    def speak(self) -> str:     # override — no @Override annotation
        return "Woof!"


class Cat(Animal):
    def speak(self) -> str:
        return "Meow!"


dog = Dog("Rex", "Labrador")
cat = Cat("Whiskers")
print(dog.speak())   # Woof!
print(cat.speak())   # Meow!
print(dog)           # Dog('Rex')

isinstance() and type checking

Python's equivalent of Java's instanceof is isinstance(). It respects the inheritance hierarchy. Prefer it over checking type(obj) == SomeClass which is too strict.

Python 
print(isinstance(dog, Dog))     # True
print(isinstance(dog, Animal))  # True  — inheritance respected
print(isinstance(cat, Dog))     # False

# Checking the exact type (usually too strict)
print(type(dog) == Dog)         # True
print(type(dog) == Animal)      # False — even though Dog IS-A Animal

Multiple inheritance

Python supports multiple inheritance — a class can inherit from several parents. Python resolves method lookup order via the MRO (Method Resolution Order), computed with the C3 linearization algorithm. You won't use this daily, but you'll encounter it in frameworks.

Python 
class Flyable:
    def fly(self) -> str:
        return "I can fly!"

class Swimmable:
    def swim(self) -> str:
        return "I can swim!"

class Duck(Animal, Flyable, Swimmable):
    def speak(self) -> str:
        return "Quack!"


donald = Duck("Donald")
print(donald.speak())  # Quack!
print(donald.fly())    # I can fly!
print(donald.swim())   # I can swim!

# Inspect the MRO (method lookup order)
print(Duck.__mro__)
# (Duck, Animal, Flyable, Swimmable, object)

Access Control Conventions

No access modifiers — conventions instead

Python has no private, protected, or public keywords. Instead, it uses naming conventions that the community respects.

Python 
class Config:
    MAX_RETRIES = 3            # public constant (ALL_CAPS)

    def __init__(self, host: str, port: int):
        self.host = host           # public — use freely
        self._port = port          # _single: "internal use, handle with care"
        self.__secret = "s3cr3t"   # __double: name-mangled

    def _validate(self):
        # _prefix: "internal helper, not part of public API"
        return self._port > 0

    def connect(self):
        # Public method — part of the API
        if self._validate():
            return f"Connecting to {self.host}:{self._port}"


c = Config("localhost", 5432)
print(c.host)      # OK — public
print(c._port)     # Works but linters/IDEs warn — you're breaking convention
# print(c.__secret)  # AttributeError! Name-mangled to _Config__secret
print(c._Config__secret)  # Still accessible if you really need it
Python's philosophy: "We're all consenting adults here." Name mangling (__dunder) prevents accidental override in subclasses — it's not true privacy. The convention is respected throughout the ecosystem.

Duck typing

In Java, you'd define an interface so a method can accept multiple types. In Python, if it has the right methods, it works — no declaration needed. This is duck typing: "If it walks like a duck and quacks like a duck, it's a duck."

Java 
interface Printable {
    void print();
}

class Report implements Printable {
    public void print() { ... }
}

void printAll(List<Printable> items) {
    for (Printable p : items) p.print();
}
Python 
class Report:
    def print(self):
        print("Printing report...")

class Invoice:
    def print(self):
        print("Printing invoice...")

class LegacyDoc:
    def print(self):
        print("Printing legacy doc...")

# No interface needed — anything with .print() works
def print_all(items):
    for item in items:
        item.print()   # works as long as item has .print()

print_all([Report(), Invoice(), LegacyDoc()])  # all work
Day 43 preview: When you need to enforce an interface (for tooling, docs, or discipline), Python has abc.ABC and typing.Protocol. Duck typing works for most cases; ABCs add rigor when needed.

Wrap-up

  • __init__ is your constructor. Every instance variable must be assigned on self inside it — there's no field declaration block like in Java.
  • Class variables (defined in the class body) are shared across instances — like Java's static fields. Assignment on self creates a new instance variable that shadows the class variable.
  • Inheritance is class Child(Parent). super() in Python 3 needs no arguments. No @Override annotation — just redefine the method.
  • Python has no access modifiers. Use _name to signal "internal" and __name for name-mangling to prevent accidental override in subclasses.
  • Duck typing means you don't need interfaces for polymorphism. If an object has the right method, it works. Day 43 covers abc.ABC and Protocol for when you want enforcement.

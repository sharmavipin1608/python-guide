1. Python's logging Module

Python has a built-in logging module. It mirrors the SLF4J / Log4j hierarchy — loggers, handlers, formatters, and log levels.

Java — SLF4J 
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class UserService {
    private static final Logger log =
        LoggerFactory.getLogger(UserService.class);

    public void createUser(String name) {
        log.info("Creating user: {}", name);
        log.debug("Debug details: {}", name);
        log.error("Something failed", exception);
    }
}
Python — logging 
import logging

# Named logger — use __name__ for module hierarchy
log = logging.getLogger(__name__)

def create_user(name: str):
    log.info("Creating user: %s", name)    # % formatting, not f-string
    log.debug("Debug details: %s", name)
    try:
        ...
    except Exception:
        log.exception("Something failed")  # logs traceback automatically

Configuration

Python 
import logging
import sys

def setup_logging(level: str = "INFO") -> None:
    logging.basicConfig(
        level=getattr(logging, level.upper()),
        format="%(asctime)s %(name)s %(levelname)s %(message)s",
        datefmt="%Y-%m-%dT%H:%M:%S",
        stream=sys.stdout,          # log to stdout (Docker-friendly)
    )
    # Silence noisy third-party loggers
    logging.getLogger("httpx").setLevel(logging.WARNING)
    logging.getLogger("sqlalchemy.engine").setLevel(logging.WARNING)

setup_logging("DEBUG")

log = logging.getLogger(__name__)
log.info("App started")
log.warning("Low disk space")
log.error("DB connection failed")

2. Structured Logging with structlog

Plain text logs are hard to query. Structured logs are JSON — every field is queryable in Datadog, CloudWatch, or Elasticsearch. structlog is the Python standard for this, similar to using logstash-logback-encoder in Java.

$ pip install structlog
Python — structlog setup 
import logging
import structlog

def configure_logging(json_logs: bool = True) -> None:
    shared_processors = [
        structlog.contextvars.merge_contextvars,     # merge request-scoped context
        structlog.stdlib.add_logger_name,
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
    ]

    if json_logs:
        # Production: emit JSON
        processors = shared_processors + [structlog.processors.JSONRenderer()]
    else:
        # Development: pretty colored output
        processors = shared_processors + [structlog.dev.ConsoleRenderer()]

    structlog.configure(
        processors=processors,
        wrapper_class=structlog.stdlib.BoundLogger,
        context_class=dict,
        logger_factory=structlog.stdlib.LoggerFactory(),
        cache_logger_on_first_use=True,
    )
    logging.basicConfig(level=logging.INFO, stream=sys.stdout)

configure_logging(json_logs=False)   # pretty in dev, True in prod

log = structlog.get_logger(__name__)

# Key-value pairs are first-class — not buried in a string
log.info("user_created", user_id=42, email="vipin@example.com")
log.warning("rate_limit_approached", requests_per_min=95, limit=100)
log.error("db_query_failed", table="tasks", duration_ms=5430, error="timeout")
JSON output (prod) 
{"event": "user_created", "user_id": 42, "email": "vipin@example.com", "level": "info", "timestamp": "2026-04-26T08:00:00Z", "logger": "app.users"}
{"event": "db_query_failed", "table": "tasks", "duration_ms": 5430, "error": "timeout", "level": "error", "timestamp": "2026-04-26T08:00:01Z"}

Bound Loggers — Persistent Context

Python 
log = structlog.get_logger()

# Bind context that appears on every subsequent log call from this logger
request_log = log.bind(user_id=42, endpoint="/tasks")
request_log.info("request_started")
request_log.info("db_queried", rows=15)
request_log.info("request_completed", duration_ms=42)
# All three lines include user_id=42 and endpoint="/tasks"

3. Correlation IDs with contextvars

A correlation ID (also called trace ID or request ID) ties together all log lines from a single request — across middleware, service calls, and Celery tasks. In Java this is MDC; in Python it's contextvars.

Java — MDC 
// In filter:
MDC.put("correlationId", requestId);

// In any service method:
log.info("Processing task");
// → {"correlationId":"abc123","message":"Processing task"}

// Cleanup:
MDC.clear();
Python — contextvars 
from contextvars import ContextVar
import structlog

# Module-level context variable
_request_id: ContextVar[str] = ContextVar("request_id", default="")

def set_request_id(request_id: str) -> None:
    _request_id.set(request_id)
    structlog.contextvars.bind_contextvars(request_id=request_id)

def get_request_id() -> str:
    return _request_id.get()
Why contextvars? Unlike a global variable, a ContextVar is isolated per asyncio task (and per thread). Setting it in one request doesn't bleed into another — exactly like MDC in Java's thread-local model.

4. FastAPI Request Logging Middleware

A middleware layer logs every request — method, path, status code, duration — and injects the correlation ID so all logs from that request are grouped.

Python — middleware 
import time
import uuid
import structlog
from fastapi import FastAPI, Request, Response
from starlette.middleware.base import BaseHTTPMiddleware
from contextvars import ContextVar

log = structlog.get_logger()
_request_id: ContextVar[str] = ContextVar("request_id", default="")

class RequestLoggingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next) -> Response:
        # Generate or read correlation ID
        request_id = request.headers.get("X-Request-ID", str(uuid.uuid4()))
        _request_id.set(request_id)
        structlog.contextvars.clear_contextvars()
        structlog.contextvars.bind_contextvars(
            request_id=request_id,
            method=request.method,
            path=request.url.path,
        )

        start = time.perf_counter()
        log.info("request_started")

        try:
            response = await call_next(request)
        except Exception:
            log.exception("request_failed")
            raise
        finally:
            duration_ms = round((time.perf_counter() - start) * 1000, 2)
            log.info(
                "request_completed",
                status_code=response.status_code,
                duration_ms=duration_ms,
            )

        response.headers["X-Request-ID"] = request_id
        return response

app = FastAPI()
app.add_middleware(RequestLoggingMiddleware)
JSON log output for one request 
{"event":"request_started","request_id":"abc-123","method":"POST","path":"/tasks","level":"info","timestamp":"2026-04-26T08:00:00Z"}
{"event":"db_queried","request_id":"abc-123","rows":1,"duration_ms":4.2,"level":"info","timestamp":"2026-04-26T08:00:00Z"}

5. OpenTelemetry Overview

OpenTelemetry (OTel) is the vendor-neutral standard for traces, metrics, and logs. It's the modern replacement for vendor-specific agents (Datadog APM, New Relic, etc.). Python has a full OTel SDK.

Python — OpenTelemetry quickstart 
# pip install opentelemetry-sdk opentelemetry-instrumentation-fastapi
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor, ConsoleSpanExporter
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor

# Setup tracer (swap ConsoleSpanExporter for OTLP exporter in prod)
provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(ConsoleSpanExporter()))
trace.set_tracer_provider(provider)

# Auto-instrument FastAPI — adds spans for every request automatically
app = FastAPI()
FastAPIInstrumentor.instrument_app(app)

# Manual spans for custom operations
tracer = trace.get_tracer(__name__)

async def process_task(task_id: int):
    with tracer.start_as_current_span("process_task") as span:
        span.set_attribute("task.id", task_id)
        # ... do work ...
        span.set_attribute("task.status", "done")
OTel vs structlog: Use both. structlog handles application-level structured logs (business events, errors). OTel handles distributed tracing (spans across services) and metrics (request rates, latencies). They complement each other — OTel can automatically inject trace IDs into your structlog context.

Quick Reference

Get logger
log = logging.getLogger(__name__)

Always use __name__ — creates logger hierarchy like package.module

Log with exception
log.exception("message")

Logs ERROR + full traceback; call inside except block

structlog logger
log = structlog.get_logger()

Returns a bound logger; add context with .bind()

Structured event
log.info("event_name", key=value, other=val)

event= is the log message; extra kwargs become JSON fields

Request context
structlog.contextvars.bind_contextvars(request_id=rid)

Bound to current async context — appears on all subsequent logs

Clear context
structlog.contextvars.clear_contextvars()

Call at request start so previous request's context doesn't leak

Silence library
logging.getLogger("httpx").setLevel(logging.WARNING)

Raise the level for noisy third-party loggers

JSON renderer
structlog.processors.JSONRenderer()

Use in prod; swap for ConsoleRenderer() in dev

Wrap-up

  • Never use print() for application logging. Use logging.getLogger(__name__) — you get log levels, filtering, and handler routing for free.
  • Structured logs (JSON) are not optional in production. They're the difference between grep-ing log files and running a SQL query against Datadog.
  • structlog.contextvars.bind_contextvars(request_id=...) is the Python equivalent of MDC — set it in middleware and it automatically appears on every log line from that request.
  • Always clear contextvars at the start of a request (clear_contextvars()), not just at the end. Async task reuse can carry context from a previous request if you only clear at the end.
  • Use structlog for application events and OpenTelemetry for distributed traces. They solve different problems and work well together.

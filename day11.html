How Decorators Work

Functions as first-class objects (recap)

Decorators are possible because Python functions are objects — they can be passed as arguments, returned from other functions, and assigned to variables. A decorator is simply a callable that takes a function and returns a (usually enhanced) function.

Python 
# Functions are objects — you can pass them around
def greet(name: str) -> str:
    return f"Hello, {name}!"

fn = greet               # assign function to variable (no call)
print(fn("Alice"))       # Hello, Alice!
print(fn.__name__)       # greet

# A function that accepts a function and returns a function
def shout(func):
    def wrapper(*args, **kwargs):
        result = func(*args, **kwargs)
        return result.upper()
    return wrapper

shouting_greet = shout(greet)
print(shouting_greet("Alice"))   # HELLO, ALICE!

The @ syntax is just shorthand

The @decorator syntax is sugar. @shout above a function definition is exactly equivalent to greet = shout(greet) after it. Nothing more.

@shout
def greet(name): ...

# is exactly the same as:
def greet(name): ...
greet = shout(greet)
Python 
def shout(func):
    def wrapper(*args, **kwargs):
        result = func(*args, **kwargs)
        return result.upper()
    return wrapper

@shout
def greet(name: str) -> str:
    return f"Hello, {name}!"

# greet is now the wrapper — the original greet is captured inside
print(greet("Alice"))   # HELLO, ALICE!

Writing Decorators

Step-by-step: a timing decorator

The standard decorator pattern always follows the same structure: outer function receives the function to wrap, defines an inner wrapper that adds behaviour before/after, and returns the wrapper.

Java — Spring AOP 
@Aspect
@Component
public class TimingAspect {

    @Around("@annotation(Timed)")
    public Object time(ProceedingJoinPoint jp)
            throws Throwable {
        long start = System.currentTimeMillis();
        Object result = jp.proceed();
        long elapsed = System.currentTimeMillis() - start;
        System.out.printf("%s took %dms%n",
            jp.getSignature().getName(), elapsed);
        return result;
    }
}
Python 
import time
import functools

def timed(func):
    """Decorator: prints how long the function took."""
    @functools.wraps(func)          # preserves func's metadata
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.4f}s")
        return result
    return wrapper


@timed
def slow_sum(n: int) -> int:
    return sum(range(n))


print(slow_sum(10_000_000))
# slow_sum took 0.3421s
# 49999995000000

functools.wraps — why it matters

Without @functools.wraps(func), the wrapped function loses its name, docstring, and signature — it looks like wrapper to any tooling, debuggers, or documentation generators. Always include it.

Python 
def bad_decorator(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper   # no @functools.wraps

def good_decorator(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper


@bad_decorator
def my_func():
    """Does something important."""
    pass

@good_decorator
def my_other_func():
    """Does something important."""
    pass

print(my_func.__name__)        # wrapper  ← WRONG
print(my_func.__doc__)         # None     ← lost!

print(my_other_func.__name__)  # my_other_func  ← correct
print(my_other_func.__doc__)   # Does something important.

Practical decorators: logging and retry

Python 
import functools
import logging

logging.basicConfig(level=logging.INFO)

def logged(func):
    """Log function entry, exit, and any exceptions."""
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        logging.info(f"Calling {func.__name__}({args}, {kwargs})")
        try:
            result = func(*args, **kwargs)
            logging.info(f"{func.__name__} returned {result!r}")
            return result
        except Exception as e:
            logging.error(f"{func.__name__} raised {type(e).__name__}: {e}")
            raise
    return wrapper


def retry(times: int = 3):
    """Retry a function up to `times` attempts on exception."""
    def decorator(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            last_exc = None
            for attempt in range(1, times + 1):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    last_exc = e
                    logging.warning(f"Attempt {attempt}/{times} failed: {e}")
            raise last_exc
        return wrapper
    return decorator   # note: three levels for parameterised decorator


@logged
def divide(a, b):
    return a / b

divide(10, 2)    # INFO: Calling divide, INFO: returned 5.0
divide(10, 0)    # ERROR: divide raised ZeroDivisionError

Decorators with Arguments

The three-layer pattern

When a decorator needs configuration — like @retry(times=5) — you add one more layer: a function that accepts the arguments and returns the actual decorator. Three layers: argument receiver → decorator → wrapper.

Python 
import functools
import time

def rate_limit(calls_per_second: float):
    """Decorator factory: limit how often a function can be called."""
    min_interval = 1.0 / calls_per_second
    last_called = {}   # mutable dict to allow closure mutation

    def decorator(func):
        last_called[func] = 0.0

        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            elapsed = time.time() - last_called[func]
            if elapsed < min_interval:
                time.sleep(min_interval - elapsed)
            last_called[func] = time.time()
            return func(*args, **kwargs)
        return wrapper
    return decorator


@rate_limit(calls_per_second=2)
def fetch_data(url: str) -> str:
    return f"data from {url}"


# Called at most 2 times per second regardless of how fast you loop
for i in range(5):
    print(fetch_data(f"https://api.example.com/{i}"))
Remember the layers:
@decorator — two layers: decorator(func) → wrapper
@decorator(arg) — three layers: decorator(arg) → actual_decorator(func) → wrapper

Stacking Multiple Decorators

You can stack decorators — they apply bottom-up (closest to the function first). The order matters: the outermost decorator runs first when the function is called.

Python 
import functools, time, logging

def timed(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"[timed] {func.__name__}: {time.perf_counter()-start:.4f}s")
        return result
    return wrapper

def logged(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        print(f"[logged] calling {func.__name__}")
        return func(*args, **kwargs)
    return wrapper

def validate_positive(func):
    @functools.wraps(func)
    def wrapper(n, *args, **kwargs):
        if n <= 0:
            raise ValueError(f"Expected positive, got {n}")
        return func(n, *args, **kwargs)
    return wrapper


# Applied bottom-up:
# validate_positive wraps compute first
# then logged wraps that
# then timed wraps everything
@timed
@logged
@validate_positive
def compute(n: int) -> int:
    return sum(range(n))


result = compute(1_000_000)
# [logged] calling compute
# [timed] compute: 0.0412s
print(result)  # 499999500000

# compute(-1)  # ValueError: Expected positive, got -1
Reading order: Decorators stack like a sandwich. @timed is the outermost slice — it runs first on call and last on return. @validate_positive is innermost — it runs just before the real function.

Built-in Decorators + @lru_cache

@staticmethod and @classmethod (recap)

These are built-in decorators you already saw on Day 8. They're implemented as descriptors, not simple wrapper functions, but the usage is the same decorator syntax.

Python 
class MathUtils:
    PI = 3.14159

    @staticmethod          # no self, no cls — pure utility
    def square(x: float) -> float:
        return x * x

    @classmethod           # receives class as first arg
    def circle_area(cls, r: float) -> float:
        return cls.PI * cls.square(r)


print(MathUtils.square(4))          # 16.0
print(MathUtils.circle_area(5))     # 78.53975

@functools.lru_cache — memoization

lru_cache caches the return value of a function for given arguments. Subsequent calls with the same arguments return the cached value instantly. LRU = Least Recently Used — it evicts old entries when the cache is full.

Java — Spring Cache 
@Service
public class FibService {

    @Cacheable("fibonacci")
    public long fib(int n) {
        if (n <= 1) return n;
        return fib(n - 1) + fib(n - 2);
    }
}
Python 
from functools import lru_cache

@lru_cache(maxsize=None)   # None = unlimited cache size
def fib(n: int) -> int:
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)


# Without cache: O(2^n) — fib(40) would take seconds
# With cache: O(n) — computed once per unique n
print(fib(100))    # 354224848179261915075 — instant

# Inspect the cache
print(fib.cache_info())
# CacheInfo(hits=98, misses=101, maxsize=None, currsize=101)

# Clear the cache if needed
fib.cache_clear()


# Python 3.9+: @cache is shorthand for @lru_cache(maxsize=None)
from functools import cache

@cache
def factorial(n: int) -> int:
    return 1 if n == 0 else n * factorial(n - 1)
Requirement: lru_cache works only with hashable arguments — all args must be usable as dict keys. Lists, dicts, and other mutables won't work. Use tuples instead.

Class-based decorators

A decorator can also be a class with __call__. This is useful when the decorator needs to maintain state across calls.

Python 
import functools

class CallCounter:
    """Decorator class that counts how many times a function is called."""

    def __init__(self, func):
        functools.update_wrapper(self, func)  # like @functools.wraps
        self.func = func
        self.call_count = 0

    def __call__(self, *args, **kwargs):
        self.call_count += 1
        return self.func(*args, **kwargs)


@CallCounter
def greet(name: str) -> str:
    return f"Hello, {name}!"


greet("Alice")
greet("Bob")
greet("Carol")
print(greet.call_count)   # 3 — state survives across calls
print(greet("Dave"))      # Hello, Dave!
print(greet.call_count)   # 4

Quick Reference Card

Basic decorator
def dec(func): / @functools.wraps(func) / def wrapper(...): ... / return wrapper

Always include @functools.wraps(func) on the inner wrapper.

Apply decorator
@dec / def fn(): ...

Equivalent to fn = dec(fn) — just syntactic sugar.

Decorator with args
def dec(arg): / def decorator(func): / def wrapper(...): ... / return wrapper / return decorator

Three layers. @dec(arg) calls dec(arg) first, which returns the real decorator.

Stack decorators
@outer / @inner / def fn(): ...

Applied bottom-up: inner wraps fn first, then outer wraps that.

Memoization
from functools import cache / @cache / def fn(n): ...

Caches results by argument. Requires hashable args. Use fn.cache_clear() to reset.

LRU cache with limit
@lru_cache(maxsize=128)

Keeps last 128 unique arg combinations. fn.cache_info() shows hits/misses.

functools.wraps
@functools.wraps(func)

Copies __name__, __doc__, __annotations__ from original to wrapper.

Class-based decorator
class Dec: / def __init__(self, func): ... / def __call__(self, ...): ...

Use when the decorator needs persistent state across calls.

Wrap-up

  • A decorator is just a function that takes a function and returns a (usually enhanced) function. The @ syntax is sugar for fn = decorator(fn). Once you see it as plain Python, there's no magic.
  • Always apply @functools.wraps(func) inside your decorator's wrapper function. Without it, the wrapped function loses its name, docstring, and signature — breaking debugging, documentation, and introspection tools.
  • Decorators with arguments need three layers: the outer function receives the arguments, the middle function receives the function to wrap, and the inner wrapper does the actual work.
  • Stacked decorators apply bottom-up at definition time. The topmost decorator is outermost at call time — think of it as the first line of defence (e.g., auth check before timing).
  • @cache / @lru_cache from functools is one-line memoization. It's the Python equivalent of Spring's @Cacheable — use it for expensive pure functions, especially recursive ones like Fibonacci.

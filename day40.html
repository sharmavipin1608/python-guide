Celery + Redis

LLM calls, email sending, image processing — anything slow doesn't belong in an HTTP request. Celery is Python's distributed task queue: offload work to background workers, track results, retry on failure. The Spring @Async equivalent, but distributed.

1. The Celery Architecture

Celery has three components: a broker (the message queue — receives task requests), workers (processes that execute tasks), and a result backend (stores task results so callers can retrieve them).

FastAPI request → task.delay() → Redis (broker) → Celery worker → executes task ↓ FastAPI polls → AsyncResult.get() ← Redis (result backend) ← task result stored
Why not just use asyncio? asyncio concurrency is single-process. Celery workers can run on multiple machines and survive your web server restarting. It also has retries, rate limiting, and scheduling built in — asyncio has none of that.

2. Setting Up Celery

celery_app.py 
from celery import Celery

# broker: where task messages are sent
# backend: where task results are stored
celery_app = Celery(
    "tasks",
    broker="redis://localhost:6379/0",
    backend="redis://localhost:6379/1",    # use a different DB for results
)

celery_app.conf.update(
    task_serializer="json",
    result_serializer="json",
    accept_content=["json"],
    timezone="UTC",
    enable_utc=True,
    task_track_started=True,               # track "started" state (not just pending/success/fail)
    task_acks_late=True,                   # only ack after task completes (safer on worker crash)
    worker_prefetch_multiplier=1,          # one task at a time per worker (good for long tasks)
)

3. Task Patterns

Basic Task + Result Retrieval

Python 
from celery_app import celery_app
from celery.result import AsyncResult

@celery_app.task
def add(x: int, y: int) -> int:
    return x + y

# Queue the task
result = add.delay(10, 20)
print(result.id)            # "a1b2c3d4-..."

# Check status (non-blocking)
print(result.state)         # PENDING → STARTED → SUCCESS / FAILURE
print(result.ready())       # False until complete

# Wait for result (blocking — use only in scripts, not FastAPI routes)
value = result.get(timeout=30)
print(value)                # 30

apply_async — Full Control

Python 
from datetime import datetime, timedelta

# Delay execution by 60 seconds
result = generate_report.apply_async(
    args=[42],
    countdown=60,
)

# Execute at a specific time
result = generate_report.apply_async(
    args=[42],
    eta=datetime.utcnow() + timedelta(hours=1),
)

# Route to a specific queue (for priority or worker specialization)
result = generate_report.apply_async(
    args=[42],
    queue="high_priority",
)

LLM Task — Real Use Case

Python 
from celery_app import celery_app
from openai import OpenAI

@celery_app.task(name="tasks.process_document")
def process_document(document_id: int, content: str) -> dict:
    """Process a document with an LLM — takes 10-30 seconds."""
    client = OpenAI()
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "Extract key points from this document."},
            {"role": "user", "content": content},
        ],
    )
    summary = response.choices[0].message.content
    # Save to DB here
    return {"document_id": document_id, "summary": summary}

4. Retries and Error Handling

Python 
import httpx
from celery_app import celery_app
from celery.utils.log import get_task_logger

logger = get_task_logger(__name__)

@celery_app.task(
    bind=True,              # gives access to self (the task instance)
    max_retries=3,
    default_retry_delay=60, # seconds between retries
    autoretry_for=(httpx.HTTPError,),   # auto-retry on these exceptions
    retry_backoff=True,     # exponential backoff: 60s, 120s, 240s
    retry_backoff_max=600,  # cap at 10 minutes
    retry_jitter=True,      # add randomness to prevent thundering herd
)
def fetch_and_process(self, url: str) -> dict:
    """Fetch a URL and process it. Retries on network errors."""
    logger.info(f"Fetching {url} (attempt {self.request.retries + 1})")
    try:
        response = httpx.get(url, timeout=10)
        response.raise_for_status()
        return {"url": url, "status": response.status_code}
    except httpx.HTTPError as exc:
        logger.warning(f"Failed to fetch {url}: {exc}")
        raise self.retry(exc=exc)   # explicit retry with the exception

# Handle task failure in the caller
result = fetch_and_process.delay("https://api.example.com/data")
try:
    data = result.get(timeout=300)
except Exception as e:
    print(f"Task failed after all retries: {e}")

5. Periodic Tasks with Celery Beat

Celery Beat is the scheduler — it sends tasks to the queue on a schedule. Think of it as a cron daemon that's integrated with your Celery setup.

celery_app.py — beat schedule 
from celery import Celery
from celery.schedules import crontab

celery_app = Celery("tasks", broker="redis://localhost:6379/0", backend="redis://localhost:6379/1")

celery_app.conf.beat_schedule = {
    # Run every 30 seconds
    "cleanup-expired-sessions": {
        "task": "tasks.cleanup_sessions",
        "schedule": 30.0,
    },
    # Every day at 8 AM UTC
    "daily-summary-email": {
        "task": "tasks.send_daily_summary",
        "schedule": crontab(hour=8, minute=0),
        "args": ("admin@example.com",),
    },
    # Every Monday at midnight
    "weekly-report": {
        "task": "tasks.generate_weekly_report",
        "schedule": crontab(day_of_week=1, hour=0, minute=0),
    },
}

@celery_app.task(name="tasks.cleanup_sessions")
def cleanup_sessions():
    # delete expired sessions from DB
    print("Cleaning up sessions")

6. FastAPI Integration

Python — async task status polling 
from fastapi import FastAPI, HTTPException
from celery.result import AsyncResult
from tasks import process_document
from pydantic import BaseModel

app = FastAPI()

class ProcessRequest(BaseModel):
    document_id: int
    content: str

class TaskStatus(BaseModel):
    task_id: str
    status: str
    result: dict | None = None

@app.post("/documents/process", status_code=202)
async def submit_document(request: ProcessRequest):
    """Queue the document for processing. Returns immediately."""
    task = process_document.delay(request.document_id, request.content)
    return {"task_id": task.id, "status": "queued"}

@app.get("/tasks/{task_id}", response_model=TaskStatus)
async def get_task_status(task_id: str):
    """Poll this endpoint to check task progress."""
    result = AsyncResult(task_id)

    if result.state == "PENDING":
        return TaskStatus(task_id=task_id, status="pending")
    elif result.state == "STARTED":
        return TaskStatus(task_id=task_id, status="running")
    elif result.state == "SUCCESS":
        return TaskStatus(task_id=task_id, status="done", result=result.get())
    elif result.state == "FAILURE":
        raise HTTPException(status_code=500, detail=str(result.info))
    return TaskStatus(task_id=task_id, status=result.state.lower())

Quick Reference

Create Celery app
Celery("name", broker="redis://localhost:6379/0")

broker = where tasks go; backend = where results go

Define task
@celery_app.task
def my_task(arg): ...

Decorated function becomes a Celery task

Queue task (fire & forget)
my_task.delay(arg)

Returns AsyncResult immediately; task runs in worker

Queue with options
my_task.apply_async(args=[arg], countdown=60)

countdown=N delays by N seconds; eta= for specific time

Check state
AsyncResult(task_id).state

PENDING → STARTED → SUCCESS or FAILURE

Auto-retry
@task(autoretry_for=(SomeError,), max_retries=3, retry_backoff=True)

Exponential backoff prevents hammering a failing service

Start worker
celery -A celery_app worker --loglevel=info

Run in separate terminal; multiple workers = more parallelism

Start scheduler
celery -A celery_app beat --loglevel=info

Runs alongside workers for periodic tasks

Monitor
celery -A celery_app flower --port=5555

Web UI for task status, worker stats, failure rates

Wrap-up

  • Return a task_id immediately from your HTTP endpoint (202 Accepted) and let the client poll /tasks/{id}. Never block an HTTP request waiting for a slow task.
  • task.delay(args) is shorthand for task.apply_async(args=[args]). Use apply_async when you need countdown, eta, or queue routing.
  • Always set max_retries and retry_backoff=True for tasks that call external services. Without a cap, a failing task can retry forever.
  • task_acks_late=True means a task is only acknowledged (removed from the queue) after it completes successfully. If the worker crashes mid-task, the task is re-queued — safer for critical work.
  • Flower is the easiest way to understand what's happening in your Celery cluster — task rates, worker status, failure breakdown. Always run it in staging.

1. Dockerfile for a FastAPI App

A Dockerfile is a recipe for building a container image. For Python, the key decisions are: which base image, how to install dependencies, and how to start the app.

A Simple Dockerfile

Dockerfile 
# Use the official slim Python image
FROM python:3.12-slim

# Set working directory inside the container
WORKDIR /app

# Copy dependency list first (Docker layer caching trick)
COPY requirements.txt .

# Install dependencies
RUN pip install --no-cache-dir -r requirements.txt

# Copy application code
COPY . .

# Expose the port uvicorn will listen on
EXPOSE 8000

# Start the app
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
Layer caching trick: Copy requirements.txt and run pip install before copying your source code. Docker caches each layer — if your code changes but requirements.txt doesn't, pip install won't re-run. This is the same trick Java Dockerfiles use with pom.xml.

requirements.txt vs pyproject.toml

For Docker builds, a plain requirements.txt (generated with pip freeze > requirements.txt or pip-compile) is the most straightforward. You can also use pip install . if you have a pyproject.toml.

requirements.txt 
fastapi==0.115.0
uvicorn[standard]==0.30.0
sqlalchemy==2.0.35
pydantic-settings==2.5.2
psycopg2-binary==2.9.9

Building and Running

$ docker build -t myapp:latest .
$ docker run -p 8000:8000 myapp:latest
$ docker run -p 8000:8000 -e DATABASE_URL=postgresql://... myapp:latest

2. Multi-Stage Builds

Some Python packages (like psycopg2, Pillow) need C compiler tools to build. You don't want those tools in your production image — they bloat the size and increase attack surface. Multi-stage builds solve this exactly like they do in Java.

Java — multi-stage 
# Stage 1: Build
FROM maven:3.9-eclipse-temurin-21 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# Stage 2: Runtime (slim)
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=build /app/target/app.jar .
CMD ["java", "-jar", "app.jar"]
Python — multi-stage 
# Stage 1: Build wheels
FROM python:3.12 AS builder
WORKDIR /build
COPY requirements.txt .
# Build binary wheels (needs gcc, etc.)
RUN pip wheel --no-cache-dir \
    --wheel-dir /wheels \
    -r requirements.txt

# Stage 2: Runtime (slim, no build tools)
FROM python:3.12-slim
WORKDIR /app
# Install pre-built wheels — no compiler needed
COPY --from=builder /wheels /wheels
RUN pip install --no-cache-dir \
    --no-index --find-links=/wheels /wheels/*.whl
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", \
     "--host", "0.0.0.0", "--port", "8000"]
Image size impact: A single-stage build with build tools can easily be 800MB+. A multi-stage slim build is typically 150–200MB for a FastAPI app. Always use multi-stage for production images.

3. .dockerignore

The .dockerignore file works exactly like .gitignore — it tells Docker what to exclude from the build context. Without it, your entire project directory (including .git, __pycache__, local .env files) gets sent to the Docker daemon.

.dockerignore 
# Python cache
__pycache__/
*.pyc
*.pyo
.pytest_cache/
.mypy_cache/

# Virtual environment — NEVER include this
.venv/
venv/

# Environment files — secrets must not be baked into the image
.env
.env.*

# Dev/test artifacts
*.egg-info/
dist/
build/
htmlcov/

# Version control
.git/
.gitignore

# IDE
.vscode/
.idea/
Critical: Never let .env files get into a Docker image. Secrets baked into an image are exposed to anyone who pulls the image. Use environment variables or secrets managers at runtime instead.

4. docker-compose: FastAPI + Postgres

docker-compose lets you define and run multi-container apps with a single YAML file. For a typical FastAPI app with a database, you need at minimum two services: your app and Postgres.

docker-compose.yml 
version: "3.9"

services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: appdb
    volumes:
      - postgres_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"          # expose to host for debugging
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U appuser -d appdb"]
      interval: 5s
      timeout: 5s
      retries: 5

  app:
    build: .
    ports:
      - "8000:8000"
    environment:
      DATABASE_URL: postgresql://appuser:secret@db:5432/appdb
      SECRET_KEY: dev-secret-key
      ENVIRONMENT: development
    depends_on:
      db:
        condition: service_healthy    # wait for db healthcheck
    volumes:
      - .:/app                        # mount code for hot reload in dev
    command: uvicorn main:app --host 0.0.0.0 --port 8000 --reload

volumes:
  postgres_data:
$ docker-compose up # start everything
$ docker-compose up -d # detached (background)
$ docker-compose logs -f app # follow app logs
$ docker-compose down # stop and remove containers
$ docker-compose down -v # also remove volumes (wipes DB)

Service Discovery

In docker-compose, services find each other by name. Your app connects to db:5432 — not localhost:5432. Docker's internal DNS resolves db to the Postgres container's IP. This is why DATABASE_URL uses @db:5432 above.

Java parallel: This is the same pattern Spring Boot developers use — a docker-compose.yml with a spring-app service and a postgres or mysql service. The depends_on + healthcheck pattern is especially important because Postgres takes a second to be ready.

5. Environment Variables in Containers

The 12-factor app principle says config should come from the environment — never hardcoded or baked into the image. Docker gives you several ways to inject environment variables.

Three Ways to Pass Config

docker-compose.yml — three approaches 
services:
  app:
    # Approach 1: inline values (fine for non-secrets)
    environment:
      ENVIRONMENT: production
      LOG_LEVEL: info

    # Approach 2: reference host env vars (value comes from shell)
    environment:
      SECRET_KEY:         # no value = read from host environment
      DATABASE_URL:

    # Approach 3: env_file (best for local dev)
    env_file:
      - .env              # reads KEY=VALUE pairs from file
      - .env.local        # local overrides (gitignored)

Reading Env Vars in Python

config.py 
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    database_url: str
    secret_key: str
    environment: str = "development"
    log_level: str = "info"

    class Config:
        env_file = ".env"          # for local dev without Docker

settings = Settings()              # reads from environment at startup
print(settings.database_url)       # postgresql://appuser:secret@db:5432/appdb
Never use ARG for secrets. Docker ARG values are baked into the image layers and visible with docker history. Always use ENV or runtime environment variables for secrets.

6. Health Checks

A health check lets Docker (and orchestrators like Kubernetes) know whether your container is actually ready to serve traffic — not just that the process started.

FastAPI Health Endpoint

main.py 
from fastapi import FastAPI
from sqlalchemy import text
from database import engine

app = FastAPI()

@app.get("/health")
async def health():
    """Health check endpoint — used by Docker and load balancers."""
    try:
        # Verify DB connection is alive
        with engine.connect() as conn:
            conn.execute(text("SELECT 1"))
        return {"status": "ok", "database": "connected"}
    except Exception as e:
        return {"status": "error", "database": str(e)}, 503

Dockerfile HEALTHCHECK Instruction

Dockerfile 
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE 8000

# Health check: curl the /health endpoint every 30s
# Start checking after 10s, fail after 3 retries
HEALTHCHECK --interval=30s --timeout=5s \
            --start-period=10s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
$ docker ps # shows health status: healthy / unhealthy / starting
$ docker inspect <container> | python3 -c "import sys,json; d=json.load(sys.stdin); print(d[0]['State']['Health'])"

Quick Reference

Base image
FROM python:3.12-slim

Use -slim for smaller images; -alpine is smaller but has compat issues with some C extensions

Layer caching
COPY requirements.txt .

Copy deps file before source code so pip install is cached

No-cache install
pip install --no-cache-dir -r requirements.txt

Avoids storing pip cache in the image layer

Start app
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

JSON array form (exec form) — preferred over shell form

Build image
docker build -t myapp:latest .

-t tags the image; . is the build context

Run container
docker run -p 8000:8000 -e KEY=val myapp

-p maps host:container port; -e sets env vars

Compose up
docker-compose up -d

Start all services detached; use logs -f to follow

Compose down
docker-compose down -v

Stop and remove containers; -v also removes named volumes

Multi-stage copy
COPY --from=builder /wheels /wheels

Copy artifacts from a previous build stage

Health check
HEALTHCHECK --interval=30s CMD curl -f http://localhost:8000/health

Docker polls this; container marked unhealthy if it fails

Service discovery
postgresql://user:pass@db:5432/mydb

Use service name (db) not localhost inside compose network

Exec into container
docker exec -it <name> bash

Open interactive shell in running container

Wrap-up

  • Copy requirements.txt and run pip install before copying source code — Docker caches layers, so unchanged deps won't reinstall on every build.
  • Multi-stage builds are the standard for production Python images: build wheels with a full image, install them into a slim runtime image. Cuts image size by 4–5x.
  • Never bake secrets into images. Use env_file locally and runtime environment variables in production. ARG values are visible in image history.
  • In docker-compose, services communicate by service name, not localhost. Your app's DATABASE_URL should use @db:5432, not @localhost:5432.
  • Health checks are essential for any real deployment. Expose a /health endpoint that verifies DB connectivity — orchestrators use it to route traffic.

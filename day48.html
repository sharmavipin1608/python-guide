Design Patterns in Python

The Gang of Four patterns are still relevant — but Python idioms implement them radically differently from Java. Singleton via module, Strategy via first-class functions, Decorator via @decorator. Less boilerplate, same intent.

1. Singleton

Java Singleton requires a private constructor and a static field. Python has three idiomatic approaches — the cleanest being that a module itself is a singleton.

Java — classic singleton 
public class Config {
    private static Config instance;
    private String dbUrl;

    private Config() {
        this.dbUrl = System.getenv("DB_URL");
    }

    public static synchronized Config getInstance() {
        if (instance == null) {
            instance = new Config();
        }
        return instance;
    }

    public String getDbUrl() { return dbUrl; }
}
Python — module as singleton 
# config.py — the module IS the singleton
import os

DB_URL = os.environ.get("DB_URL", "sqlite:///./dev.db")
SECRET_KEY = os.environ.get("SECRET_KEY", "dev-key")

# Every import of this module gets the SAME module object
# from config import DB_URL  ← always the same value
Python — @lru_cache singleton (for class instances) 
from functools import lru_cache
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    db_url: str = "sqlite:///./dev.db"
    secret_key: str = "dev"

@lru_cache(maxsize=1)
def get_settings() -> Settings:
    return Settings()   # constructed once, cached forever

# Every call returns the same instance
s1 = get_settings()
s2 = get_settings()
assert s1 is s2   # True

2. Factory

Java's Factory pattern involves abstract factory classes, factory methods, and often a registry. Python replaces this with plain functions and dictionaries.

Java — factory method pattern 
public abstract class Exporter {
    public static Exporter create(String format) {
        return switch (format) {
            case "csv"  -> new CsvExporter();
            case "json" -> new JsonExporter();
            default -> throw new IllegalArgumentException(format);
        };
    }
    public abstract String export(List<?> data);
}
Python — function factory 
from typing import Protocol
import json, csv, io

class Exporter(Protocol):
    def export(self, data: list) -> str: ...

class CsvExporter:
    def export(self, data: list) -> str:
        out = io.StringIO()
        writer = csv.writer(out)
        for row in data:
            writer.writerow(row if isinstance(row, list) else [row])
        return out.getvalue()

class JsonExporter:
    def export(self, data: list) -> str:
        return json.dumps(data, indent=2)

_EXPORTERS: dict[str, type] = {
    "csv": CsvExporter,
    "json": JsonExporter,
}

def create_exporter(format: str) -> Exporter:
    cls = _EXPORTERS.get(format)
    if not cls:
        raise ValueError(f"Unknown format: {format!r}. Choose from {list(_EXPORTERS)}")
    return cls()

exporter = create_exporter("json")
print(exporter.export([1, 2, 3]))

3. Strategy

Java Strategy requires an interface and a class for each strategy. Python uses first-class functions — pass a callable directly. No interface, no class, no boilerplate.

Java — Strategy pattern 
@FunctionalInterface
interface SortStrategy {
    List<Task> sort(List<Task> tasks);
}

class TaskSorter {
    private SortStrategy strategy;
    TaskSorter(SortStrategy strategy) {
        this.strategy = strategy;
    }
    List<Task> sort(List<Task> tasks) {
        return strategy.sort(tasks);
    }
}

// Usage:
new TaskSorter(tasks -> tasks.stream()
    .sorted(Comparator.comparing(Task::getDueDate))
    .collect(toList()));
Python — callable strategy 
from typing import Callable
from dataclasses import dataclass
from datetime import date

@dataclass
class Task:
    title: str
    priority: int
    due_date: date

SortStrategy = Callable[[list[Task]], list[Task]]

class TaskSorter:
    def __init__(self, strategy: SortStrategy):
        self._strategy = strategy

    def sort(self, tasks: list[Task]) -> list[Task]:
        return self._strategy(tasks)

# Strategies are just functions — no class, no interface
by_due_date   = lambda tasks: sorted(tasks, key=lambda t: t.due_date)
by_priority   = lambda tasks: sorted(tasks, key=lambda t: t.priority, reverse=True)
by_title      = lambda tasks: sorted(tasks, key=lambda t: t.title.lower())

# Swap strategy at runtime
sorter = TaskSorter(by_priority)
sorter.sort(tasks)

sorter._strategy = by_due_date   # swap the strategy

4. Observer

Observer (event/listener) in Java requires an interface and explicit registration. Python uses plain callables — functions, lambdas, or methods — registered in a list.

Python — simple event bus 
from typing import Callable
from collections import defaultdict
from dataclasses import dataclass

@dataclass
class TaskEvent:
    task_id: int
    event_type: str    # "created", "completed", "deleted"
    data: dict

EventHandler = Callable[[TaskEvent], None]

class EventBus:
    def __init__(self):
        self._handlers: dict[str, list[EventHandler]] = defaultdict(list)

    def subscribe(self, event_type: str, handler: EventHandler) -> None:
        self._handlers[event_type].append(handler)

    def publish(self, event: TaskEvent) -> None:
        for handler in self._handlers.get(event.event_type, []):
            handler(event)
        for handler in self._handlers.get("*", []):   # wildcard
            handler(event)

bus = EventBus()

# Handlers are plain functions — no interface required
def send_email_notification(event: TaskEvent) -> None:
    print(f"Email: Task {event.task_id} was {event.event_type}")

def log_event(event: TaskEvent) -> None:
    print(f"Log: {event}")

def update_metrics(event: TaskEvent) -> None:
    print(f"Metrics: increment tasks.{event.event_type}")

bus.subscribe("completed", send_email_notification)
bus.subscribe("*", log_event)          # receives all events
bus.subscribe("created", update_metrics)

# Publish an event
bus.publish(TaskEvent(task_id=42, event_type="completed", data={}))
# → Email: Task 42 was completed
# → Log: TaskEvent(task_id=42, event_type='completed', ...)

5. Builder

Java Builder is verbose — a static inner Builder class with a field for each attribute. Python's @dataclass with defaults and replace() covers most use cases. For complex objects, method chaining works cleanly.

Python — Builder via dataclass + replace 
from dataclasses import dataclass, field, replace
from datetime import date

@dataclass
class QueryBuilder:
    table: str
    conditions: list[str] = field(default_factory=list)
    columns: list[str] = field(default_factory=lambda: ["*"])
    limit: int | None = None
    order_by: str | None = None

    def where(self, condition: str) -> "QueryBuilder":
        return replace(self, conditions=[*self.conditions, condition])

    def select(self, *cols: str) -> "QueryBuilder":
        return replace(self, columns=list(cols))

    def take(self, n: int) -> "QueryBuilder":
        return replace(self, limit=n)

    def order(self, col: str) -> "QueryBuilder":
        return replace(self, order_by=col)

    def build(self) -> str:
        cols = ", ".join(self.columns)
        sql = f"SELECT {cols} FROM {self.table}"
        if self.conditions:
            sql += " WHERE " + " AND ".join(self.conditions)
        if self.order_by:
            sql += f" ORDER BY {self.order_by}"
        if self.limit:
            sql += f" LIMIT {self.limit}"
        return sql

query = (
    QueryBuilder("tasks")
    .select("id", "title", "status")
    .where("status = 'pending'")
    .where("priority > 1")
    .order("due_date")
    .take(20)
    .build()
)
print(query)
# SELECT id, title, status FROM tasks WHERE status = 'pending' AND priority > 1 ORDER BY due_date LIMIT 20

6. Decorator Pattern vs Python @decorator

These are two different things with the same name. The GoF Decorator pattern wraps objects to add behaviour. Python's @decorator syntax wraps functions. Python's syntax implements the pattern, but more elegantly.

Python — GoF Decorator pattern (object wrapping) 
from typing import Protocol

class TextProcessor(Protocol):
    def process(self, text: str) -> str: ...

class PlainProcessor:
    def process(self, text: str) -> str:
        return text

class UpperCaseDecorator:
    def __init__(self, wrapped: TextProcessor):
        self._wrapped = wrapped
    def process(self, text: str) -> str:
        return self._wrapped.process(text).upper()

class TrimDecorator:
    def __init__(self, wrapped: TextProcessor):
        self._wrapped = wrapped
    def process(self, text: str) -> str:
        return self._wrapped.process(text).strip()

# Stack decorators at runtime
processor = TrimDecorator(UpperCaseDecorator(PlainProcessor()))
print(processor.process("  hello world  "))   # "HELLO WORLD"
Python — same result with function decorators 
import functools

def uppercase(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs).upper()
    return wrapper

def trim(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs).strip()
    return wrapper

@trim
@uppercase
def process(text: str) -> str:
    return text

print(process("  hello world  "))   # "HELLO WORLD"
# Decorators apply bottom-up: uppercase first, then trim

Quick Reference

Singleton
Module-level state OR @lru_cache on a factory function

A module is already a singleton — no class needed

Factory
def create(kind: str): return _REGISTRY[kind]()

Dict of type → class; return cls() to instantiate

Strategy
Callable[[Input], Output]

Pass a function; swap at runtime — no interface class needed

Observer subscribe
bus.subscribe("event", handler_fn)

Handlers are plain callables — lambdas, methods, or functions

Builder via replace
from dataclasses import replace
new = replace(old, field=value)

Returns a new instance with specified fields changed — immutable builder

Stack decorators
@outer
@inner
def fn(): ...

Apply bottom-up: inner wraps fn, then outer wraps that

Python idiom rule
# If you're writing a Java pattern verbatim, pause

Most GoF patterns collapse to functions, dicts, or decorators in Python

functools.wraps
@functools.wraps(func)

Preserves __name__, __doc__ on the wrapper function

Wrap-up

  • In Python, a module is already a singleton — importing it twice returns the same object. Reach for the module-level singleton before writing a Singleton class.
  • Strategy in Java requires an interface + a class per strategy. In Python, pass a function. Callable[[T], R] is your interface, and any matching function is a valid implementation.
  • The Observer pattern collapses to a list of callables. A dict of event_type → list[handler] is a complete, practical event bus — no framework needed.
  • dataclasses.replace() gives you an immutable builder pattern: each method returns a new object with one field changed. Chains read left-to-right and nothing is mutated in place.
  • When you find yourself writing a Java pattern verbatim in Python — multiple classes, lots of ceremony — pause and ask if a function, dict, or decorator achieves the same intent with less code.

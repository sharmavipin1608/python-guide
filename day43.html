Advanced OOP: ABCs + Protocols

Python has two distinct ways to define contracts: ABCs enforce nominal typing (like Java interfaces), and Protocols enforce structural typing (duck typing with type checker support). Knowing when to use each is a senior Python skill.

What You'll Cover

  • abc.ABC and @abstractmethod — enforced nominal contracts ~20 min

Java Parallels

  • abc.ABC + @abstractmethod → Java abstract class or interface with enforced implementation
  • typing.Protocol → structural interface — like if Java matched types by shape, not by implements
  • ABC → nominal typing — you must explicitly inherit from it
  • Protocol → structural typing — if it has the right methods, it satisfies the contract
  • runtime_checkable Protocol → limited isinstance() check by shape, not inheritance

1. Why Python Doesn't Need Interfaces (Usually)

In Java, interfaces are the primary mechanism for polymorphism — you can't pass a List where a Collection is expected without implementing the interface. Python uses duck typing: if an object has the right methods, it works, regardless of its class hierarchy.

Java — interface required 
interface Serializable {
    String serialize();
}

class JsonSerializer implements Serializable {
    public String serialize() { return "{}"; }
}

// Must declare implements Serializable
void save(Serializable s) {
    System.out.println(s.serialize());
}
Python — duck typing 
class JsonSerializer:
    def serialize(self) -> str:
        return "{}"

class XmlSerializer:
    def serialize(self) -> str:
        return ""

# No interface needed — any object with .serialize() works
def save(s) -> None:
    print(s.serialize())

save(JsonSerializer())   # works
save(XmlSerializer())    # works too
Duck typing is fine for small codebases. But as a project grows, you lose the ability to know what a function actually requires. ABCs and Protocols solve this — they bring back contracts without requiring Java-style explicit declarations.

2. abc.ABC — Enforced Abstract Contracts

Inherit from ABC and decorate methods with @abstractmethod. Python will refuse to instantiate any subclass that hasn't implemented every abstract method — enforced at runtime, not compile time.

Java — abstract class 
public abstract class Repository<T> {
    public abstract T findById(int id);
    public abstract void save(T entity);
    public abstract void delete(int id);
    public abstract List<T> findAll();

    // Concrete method in abstract class
    public boolean exists(int id) {
        return findById(id) != null;
    }
}
Python — ABC 
from abc import ABC, abstractmethod
from typing import Generic, TypeVar

T = TypeVar("T")

class Repository(ABC, Generic[T]):
    @abstractmethod
    def find_by_id(self, id: int) -> T | None: ...

    @abstractmethod
    def save(self, entity: T) -> None: ...

    @abstractmethod
    def delete(self, id: int) -> None: ...

    @abstractmethod
    def find_all(self) -> list[T]: ...

    # Concrete method is fine in an ABC
    def exists(self, id: int) -> bool:
        return self.find_by_id(id) is not None

Implementing an ABC

Python 
from dataclasses import dataclass

@dataclass
class Task:
    id: int
    title: str

class InMemoryTaskRepository(Repository[Task]):
    def __init__(self):
        self._store: dict[int, Task] = {}

    def find_by_id(self, id: int) -> Task | None:
        return self._store.get(id)

    def save(self, entity: Task) -> None:
        self._store[entity.id] = entity

    def delete(self, id: int) -> None:
        self._store.pop(id, None)

    def find_all(self) -> list[Task]:
        return list(self._store.values())

# Works fine
repo = InMemoryTaskRepository()
repo.save(Task(id=1, title="Buy milk"))

# Missing an abstract method? Python raises TypeError at instantiation:
class BrokenRepo(Repository[Task]):
    pass  # forgot to implement methods

# BrokenRepo()  → TypeError: Can't instantiate abstract class BrokenRepo
#                  with abstract methods delete, find_all, find_by_id, save

Abstract Properties and Class Methods

Python 
from abc import ABC, abstractmethod

class DataSource(ABC):
    @property
    @abstractmethod
    def connection_string(self) -> str:
        """Subclasses must implement this property."""
        ...

    @classmethod
    @abstractmethod
    def from_env(cls) -> "DataSource":
        """Factory method that reads config from environment."""
        ...

class PostgresDataSource(DataSource):
    @property
    def connection_string(self) -> str:
        return "postgresql://localhost/mydb"

    @classmethod
    def from_env(cls) -> "PostgresDataSource":
        import os
        return cls()  # in reality, reads from os.environ

3. typing.Protocol — Structural Typing

A Protocol defines a contract by shape — any class that has the required methods and attributes satisfies the Protocol, without explicitly inheriting from it. This is structural typing, and it's what mypy/pyright use to check duck-typed code.

Python — Protocol 
from typing import Protocol, runtime_checkable

@runtime_checkable   # allows isinstance() checks
class Serializable(Protocol):
    def serialize(self) -> str: ...
    def deserialize(self, data: str) -> None: ...

# Neither class inherits from Serializable
class JsonSerializer:
    def serialize(self) -> str:
        return '{"type": "json"}'
    def deserialize(self, data: str) -> None:
        import json; json.loads(data)

class XmlSerializer:
    def serialize(self) -> str:
        return ""
    def deserialize(self, data: str) -> None:
        pass   # imagine real XML parsing here

# Type-safe function — mypy verifies the argument matches Serializable
def process(s: Serializable) -> str:
    return s.serialize()

process(JsonSerializer())   # ✅ type checker is happy
process(XmlSerializer())    # ✅ also fine — both satisfy the Protocol

# runtime check (only works with @runtime_checkable)
print(isinstance(JsonSerializer(), Serializable))  # True

Protocol with Attributes

Python 
from typing import Protocol

class HasName(Protocol):
    name: str      # attribute requirement

class HasId(Protocol):
    id: int

class Identifiable(HasName, HasId, Protocol):
    """Composed Protocol — combines multiple smaller ones."""
    ...

# Any class with .name: str and .id: int satisfies Identifiable
from dataclasses import dataclass

@dataclass
class User:
    id: int
    name: str
    email: str   # extra fields are fine

@dataclass
class Product:
    id: int
    name: str
    price: float

def display(item: Identifiable) -> str:
    return f"[{item.id}] {item.name}"

display(User(id=1, name="Vipin", email="v@e.com"))      # ✅
display(Product(id=42, name="Widget", price=9.99))       # ✅

4. ABC vs Protocol — When to Use Each

SituationUse ABCUse Protocol
You own all implementations✅ Good fit✅ Also fine
Third-party classes need to satisfy the contract❌ They can't inherit your ABC✅ Perfect — no changes needed
You want shared implementation (template method pattern)✅ ABC supports concrete methods❌ Protocol is interface only
Enforced at runtime (instantiation check)✅ TypeError if abstract methods missing⚠️ Only with @runtime_checkable + isinstance
Static type checking only✅✅
Working with duck-typed ecosystem code❌ Must inherit✅ Works transparently
Rule of thumb: Use Protocol for type-checking function parameters ("I need something that can be serialized"). Use ABC when you have shared behavior to provide and you control all implementations ("I need a repository with a common exists() method").

5. Real-world: Repository Pattern with Protocol

Python 
from typing import Protocol
from dataclasses import dataclass

@dataclass
class Task:
    id: int
    title: str
    done: bool = False

# Protocol — any repository implementation satisfies this
class TaskRepository(Protocol):
    def get(self, task_id: int) -> Task | None: ...
    def list_all(self) -> list[Task]: ...
    def save(self, task: Task) -> None: ...
    def delete(self, task_id: int) -> None: ...

# Implementation 1 — for production
class SQLTaskRepository:
    def __init__(self, db_session): self.db = db_session
    def get(self, task_id: int) -> Task | None: ...    # SQLAlchemy query
    def list_all(self) -> list[Task]: ...
    def save(self, task: Task) -> None: ...
    def delete(self, task_id: int) -> None: ...

# Implementation 2 — for tests (no DB needed)
class InMemoryTaskRepository:
    def __init__(self): self._store: dict[int, Task] = {}
    def get(self, task_id: int) -> Task | None:
        return self._store.get(task_id)
    def list_all(self) -> list[Task]:
        return list(self._store.values())
    def save(self, task: Task) -> None:
        self._store[task.id] = task
    def delete(self, task_id: int) -> None:
        self._store.pop(task_id, None)

# Service uses the Protocol — works with either implementation
class TaskService:
    def __init__(self, repo: TaskRepository):   # typed to Protocol
        self.repo = repo

    def complete_task(self, task_id: int) -> bool:
        task = self.repo.get(task_id)
        if not task:
            return False
        task.done = True
        self.repo.save(task)
        return True

# In production:
# service = TaskService(repo=SQLTaskRepository(db_session))

# In tests:
in_memory = InMemoryTaskRepository()
service = TaskService(repo=in_memory)   # no DB needed!
in_memory.save(Task(id=1, title="Buy milk"))
assert service.complete_task(1) is True
assert in_memory.get(1).done is True

Wrap-up

  • ABCs enforce nominal contracts — subclasses must explicitly inherit and implement every abstract method. Python raises TypeError at instantiation time if they don't.
  • Protocols enforce structural contracts — any class with the right methods satisfies it, regardless of its class hierarchy. This is how you type-check duck-typed code.
  • Use Protocol when you don't control all implementations (third-party code, test doubles). Use ABC when you want to share concrete method implementations across subclasses.
  • The repository pattern is the canonical use case: define a TaskRepository Protocol, swap in SQLTaskRepository for production and InMemoryTaskRepository for tests — zero changes to service code.
  • @runtime_checkable enables isinstance() checks against a Protocol, but it only verifies that methods/attributes exist — not their signatures. Rely on mypy/pyright for full type safety.

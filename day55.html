1. Pythonic Code Checklist

Read through this list honestly. Check each item against code you've written over the past 54 days. If something still feels foreign, spend 15 minutes re-reading the relevant day.

Variables + Data Structures

  • I use _ for throwaway variables: for _, value in pairs:
  • I unpack tuples directly: x, y = point instead of point[0], point[1]
  • I use dict.get(key, default) instead of checking if key in dict then accessing
  • I reach for collections.defaultdict and Counter instead of manual counting dicts
  • I use set for membership tests and deduplication, not list
  • I use dataclasses or NamedTuple instead of bare dicts for structured data

Loops + Comprehensions

  • I iterate directly over collections, not over range(len(...))
  • I use enumerate() when I need both index and value
  • I use zip() to iterate over multiple sequences in parallel
  • I write list comprehensions for simple transformations and filters
  • I use generator expressions (x for x in ...) when I only need to iterate once
  • I don't put side effects inside comprehensions

Functions + OOP

  • My functions do one thing and have descriptive names (snake_case)
  • I use *args and **kwargs where appropriate
  • I use @property instead of Java-style get_x() / set_x() methods
  • My classes use __str__ for human-readable output, __repr__ for debugging
  • I write @dataclass instead of classes that only hold data with manual __init__
  • I use Protocol for structural typing instead of requiring inheritance

Errors + Context

  • I catch specific exceptions, not bare except:
  • I use with statements for resources (files, DB connections, locks)
  • I raise meaningful custom exceptions with informative messages
  • I use logging, not print(), for application output

Type Hints + Style

  • I add type hints to all public function signatures
  • I use X | None (Python 3.10+) instead of Optional[X]
  • I use str | None = None for optional parameters
  • My files pass ruff check with no errors

2. Common Anti-Patterns

These are the patterns that scream "Java developer learning Python". If you spot any in your code, refactor them now — before the capstone.

Anti-pattern 1: Java-style getters and setters 
# ❌ Java-in-Python
class Task:
    def get_title(self): return self._title
    def set_title(self, value): self._title = value

# ✅ Pythonic
class Task:
    @property
    def title(self) -> str: return self._title
    @title.setter
    def title(self, value: str): self._title = value
    # Or just: self.title = value  if no validation needed
Anti-pattern 2: Indexing when you should iterate 
# ❌ Java-in-Python
for i in range(len(tasks)):
    print(tasks[i].title)

# ✅ Pythonic
for task in tasks:
    print(task.title)

# ✅ When you need the index
for i, task in enumerate(tasks):
    print(f"{i}: {task.title}")
Anti-pattern 3: Type checking instead of duck typing 
# ❌ Java-in-Python
def process(obj):
    if isinstance(obj, list):
        for item in obj:
            do_thing(item)
    elif isinstance(obj, tuple):
        for item in obj:
            do_thing(item)

# ✅ Pythonic — if it's iterable, iterate it
def process(obj):
    for item in obj:   # works for list, tuple, set, generator, etc.
        do_thing(item)
Anti-pattern 4: Building strings with concatenation 
# ❌ Java-in-Python — O(n²) memory allocations
result = ""
for item in items:
    result += str(item) + ", "

# ✅ Pythonic
result = ", ".join(str(item) for item in items)

# ✅ For complex formatting, use f-strings
name, score = "Vipin", 95
msg = f"User {name} scored {score}/100"
Anti-pattern 5: Returning None implicitly and ignoring it 
# ❌ Java-in-Python — caller has no idea this can return None
def find_user(user_id: int):
    for u in users:
        if u.id == user_id:
            return u
    # implicit return None — undocumented!

# ✅ Pythonic — explicit return type, caller knows to handle None
def find_user(user_id: int) -> User | None:
    return next((u for u in users if u.id == user_id), None)
Anti-pattern 6: Mutable default arguments 
# ❌ Bug — the same list is shared across ALL calls
def add_item(item, items=[]):
    items.append(item)
    return items

print(add_item("a"))  # ['a']
print(add_item("b"))  # ['a', 'b']  ← BUG: list persists between calls

# ✅ Pythonic
def add_item(item: str, items: list[str] | None = None) -> list[str]:
    if items is None:
        items = []
    items.append(item)
    return items

3. Read a Real Python Codebase

The best way to internalize idiomatic Python is to read code written by experienced Python developers. Pick one of these well-maintained open-source projects and spend 20–30 minutes exploring:

FastAPI

The framework you've been using. Read the routing, dependency injection, and middleware source. Excellent async Python and type hint usage.

github.com/fastapi/fastapi

httpx

A modern HTTP client. Clean API design, excellent use of context managers, sync + async dual support in one codebase.

github.com/encode/httpx

Pydantic

If you want to see metaclasses, descriptors, and Python's data model used in anger — the Pydantic v2 source is fascinating.

github.com/pydantic/pydantic

rich

Beautiful terminal output library. Great example of a well-structured Python library with clean public API and extensive use of dataclasses and Protocols.

github.com/Textualize/rich

attrs

The predecessor to dataclasses. Compact, readable source that shows advanced Python class manipulation.

github.com/python-attrs/attrs

click

The CLI framework from Day 47. Well-documented source, excellent decorator-heavy design.

github.com/pallets/click
What to look for when reading: How they structure modules. How they handle errors. Where they use type hints. What design patterns they use and which Java patterns they've replaced with simpler Python idioms. Notice what you understand immediately vs. what still looks unfamiliar.

Quick Reference — Pythonic One-Liners

Safe dict access
value = d.get(key, default)

Never KeyError; returns default if key missing

First match
next((x for x in items if cond(x)), None)

Returns first match or None — no StopIteration

Flatten a list
[x for sub in nested for x in sub]

Or: itertools.chain.from_iterable(nested)

Group by key
from collections import defaultdict
d = defaultdict(list)
for item in items: d[item.key].append(item)

No KeyError, no if-else boilerplate

Swap variables
a, b = b, a

No temp variable needed

Conditional expression
value = x if condition else y

Python's ternary — reads left to right

Merge dicts
merged = {**dict1, **dict2}

Or: dict1 | dict2 (Python 3.9+)

Check all/any
all(t.done for t in tasks)
any(t.done for t in tasks)

Short-circuits — more idiomatic than a loop

Wrap-up

  • The shift from Java to Python is not just syntax — it's moving from "explicit structure" to "trust the idioms". The more you reach for comprehensions, generators, Protocols, and context managers, the more Pythonic your code becomes.
  • Reading real codebases accelerates learning faster than any tutorial. FastAPI's source code alone contains more Python idioms than most books.
  • The most persistent Java-ism in Python code is excessive class creation. Python functions are first-class objects — a function is often the right level of abstraction, not a class.

1. timeit — Micro-benchmarks

Always start with timeit when comparing two implementations. It runs the expression many times and accounts for startup overhead.

Python — timeit in code 
import timeit

# Compare string concatenation approaches
concat_loop = timeit.timeit(
    stmt="""
s = ""
for i in range(1000):
    s += str(i)
""",
    number=1000,
)

join_approach = timeit.timeit(
    stmt="""s = "".join(str(i) for i in range(1000))""",
    number=1000,
)

print(f"Loop concat:  {concat_loop:.4f}s")
print(f"join():       {join_approach:.4f}s")
# join() is typically 3-5x faster for large strings

# Compare list vs set for 'in' lookups
data_list = list(range(10_000))
data_set  = set(range(10_000))

list_lookup = timeit.timeit("9999 in data_list", globals=globals(), number=100_000)
set_lookup  = timeit.timeit("9999 in data_set",  globals=globals(), number=100_000)
print(f"list 'in': {list_lookup:.4f}s")
print(f"set 'in':  {set_lookup:.4f}s")
# Set is O(1), list is O(n) — set wins dramatically for large collections
$ python -m timeit -n 10000 '"".join(str(i) for i in range(100))'
$ python -m timeit -s "data = list(range(10000))" "9999 in data"

2. cProfile — Function-Level Profiling

cProfile is built into Python's stdlib. It records how many times each function is called and how long each call takes — the right first tool when you need to find which function is the bottleneck.

Python — cProfile in code 
import cProfile
import pstats
import io

def slow_function():
    return sum(i * i for i in range(500_000))

def medium_function():
    return [x ** 2 for x in range(100_000)]

def entry_point():
    slow_function()
    medium_function()
    medium_function()

# Profile and capture output
profiler = cProfile.Profile()
profiler.enable()
entry_point()
profiler.disable()

# Print sorted by cumulative time
stream = io.StringIO()
stats = pstats.Stats(profiler, stream=stream)
stats.sort_stats("cumulative")
stats.print_stats(10)   # top 10 functions
print(stream.getvalue())
$ # Profile a script from the command line
$ python -m cProfile -s cumulative myscript.py

$ # Save to file for snakeviz visualization
$ python -m cProfile -o profile.out myscript.py
$ pip install snakeviz
$ snakeviz profile.out # opens interactive flame graph in browser

# percall — time per single call

3. line_profiler — Line-by-Line

Once cProfile tells you which function is slow, line_profiler tells you which line inside it. Decorate target functions with @profile and run with kernprof.

$ pip install line_profiler
Python — line_profiler usage 
# Decorate the function you want to profile
@profile   # injected by kernprof at runtime — no import needed
def process_data(records: list[dict]) -> list[dict]:
    results = []
    for record in records:                           # line 4
        cleaned = {k: v.strip() for k, v in record.items() if isinstance(v, str)}  # line 5
        if cleaned.get("status") == "active":        # line 6
            cleaned["score"] = sum(                  # line 7
                len(v) for v in cleaned.values()
            )
            results.append(cleaned)                  # line 9
    return results

4. memory_profiler — Memory Usage

$ pip install memory_profiler
Python 
from memory_profiler import profile

@profile
def load_and_process(filepath: str) -> list:
    with open(filepath) as f:
        data = f.readlines()          # loads all lines into memory at once
    processed = [line.strip().upper() for line in data]
    return processed

# Run: python -m memory_profiler myscript.py
# Output:
# Line #  Mem usage    Increment   Line Contents
# ================================================
#      5   45.3 MiB    45.3 MiB   with open(filepath) as f:
#      6  125.8 MiB    80.5 MiB       data = f.readlines()   ← huge allocation
#      7  210.1 MiB    84.3 MiB   processed = [line.strip()...]
#      8  125.8 MiB   -84.3 MiB   return processed           ← original freed

Generator Fix for Memory-Heavy Code

Python 
@profile
def load_and_process_lazy(filepath: str):
    """Process line by line — constant memory regardless of file size."""
    with open(filepath) as f:
        for line in f:             # file object is a lazy iterator
            yield line.strip().upper()

# Memory stays flat — no large allocation spike
# The trade-off: you can only iterate once; no random access

5. Common Python Performance Pitfalls

Python — pitfalls and fixes 
## 1. String concatenation in a loop
# SLOW — creates a new string on every iteration O(n²)
result = ""
for word in words:
    result += word + " "

# FAST — join allocates once
result = " ".join(words)


## 2. 'in' on a list vs set
# SLOW — O(n) linear scan
if item in my_list:      # 10,000 items = 10,000 comparisons worst case
    ...

# FAST — O(1) hash lookup
lookup_set = set(my_list)
if item in lookup_set:
    ...


## 3. Repeated attribute lookup in a tight loop
import math
# SLOW — looks up math.sqrt in the module dict every iteration
for x in data:
    result = math.sqrt(x)

# FAST — cache the reference
sqrt = math.sqrt
for x in data:
    result = sqrt(x)


## 4. Building a list when you only need iteration
# SLOW — materialises the whole list in memory
for item in list(my_generator):
    process(item)

# FAST — iterate the generator directly
for item in my_generator:
    process(item)


## 5. Unnecessary function call overhead
# SLOW — calling len() on each loop iteration
for i in range(len(my_list)):
    ...

# FAST — iterate directly
for item in my_list:
    ...

# Or if you need the index:
for i, item in enumerate(my_list):
    ...


## 6. dict.get() vs key check
# SLOW — two hash lookups
if key in d:
    value = d[key]

# FAST — one hash lookup
value = d.get(key)    # returns None if missing

# Or with defaultdict for counting
from collections import defaultdict
counts = defaultdict(int)
for word in words:
    counts[word] += 1   # no KeyError, no 'if word in counts' check

When to Reach for NumPy / C Extensions

Python 
import numpy as np
import timeit

data = list(range(1_000_000))
arr = np.array(data)

# Python loop — slow, GIL overhead per iteration
python_sum = timeit.timeit(lambda: sum(x*x for x in data), number=10)

# NumPy — vectorised C operation, GIL released
numpy_sum = timeit.timeit(lambda: np.sum(arr ** 2), number=10)

print(f"Python: {python_sum:.3f}s")
print(f"NumPy:  {numpy_sum:.3f}s")
# NumPy is typically 50-200x faster for numerical work

# Rule: if you're looping over numbers in Python, consider NumPy
# If you're looping over heterogeneous data (dicts, strings), NumPy won't help

Wrap-up

  • Always measure before optimising. Use timeit for quick comparisons, cProfile to find hot functions, line_profiler to find the hot line within those functions.
  • The single biggest win in most Python code: replace list with set for membership tests. O(n) → O(1) is often a 100–1000× speedup for large collections.
  • String concatenation with += in a loop is O(n²) — Python creates a new string object each time. Use "".join(parts) instead.
  • For numerical loops, NumPy is the correct answer — not micro-optimising Python loops. A NumPy vectorised operation can be 100× faster because it runs compiled C with the GIL released.
  • Generators are the memory-efficient alternative to building large lists. If you only need to iterate once, yield instead of accumulating — memory usage stays constant regardless of input size.

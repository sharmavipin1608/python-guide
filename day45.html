1. The GIL — Global Interpreter Lock

The GIL is a mutex that allows only one thread to execute Python bytecode at a time within a single CPython process. It's the most misunderstood aspect of Python for developers coming from Java.

JVM (Java): Thread 1 ──── CPU core 1 ┐ Thread 2 ──── CPU core 2 ├─ true parallel CPU execution Thread 3 ──── CPU core 3 ┘ CPython (Python — GIL present): Thread 1 ─┐ Thread 2 ─┤─ GIL ──── CPU core 1 ← only one at a time Thread 3 ─┘ (threads take turns holding the GIL every ~5ms)
Python — GIL impact on CPU-bound work 
import threading
import time

def cpu_bound(n: int) -> int:
    """Pure Python CPU work — affected by GIL."""
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 50_000_000

# Single thread
start = time.perf_counter()
cpu_bound(N)
single_time = time.perf_counter() - start

# Two threads — you'd expect 2x speedup; you won't get it
start = time.perf_counter()
t1 = threading.Thread(target=cpu_bound, args=(N // 2,))
t2 = threading.Thread(target=cpu_bound, args=(N // 2,))
t1.start(); t2.start()
t1.join(); t2.join()
thread_time = time.perf_counter() - start

print(f"Single: {single_time:.2f}s")
print(f"Two threads: {thread_time:.2f}s")
# Two threads is roughly the SAME as single — GIL prevents true parallel CPU work

What the GIL Does NOT Block

Python — I/O releases the GIL 
import threading
import urllib.request
import time

def fetch(url: str) -> None:
    # During network I/O, the GIL is released — other threads run freely
    urllib.request.urlopen(url).read()

urls = ["https://httpbin.org/delay/1"] * 5

# Sequential: 5 seconds
start = time.perf_counter()
for url in urls:
    fetch(url)
print(f"Sequential: {time.perf_counter() - start:.1f}s")   # ~5s

# Threaded: ~1 second — GIL released during I/O
start = time.perf_counter()
threads = [threading.Thread(target=fetch, args=(url,)) for url in urls]
for t in threads: t.start()
for t in threads: t.join()
print(f"Threaded: {time.perf_counter() - start:.1f}s")    # ~1s
GIL summary: Threads are fine for I/O-bound work (network, file, DB). For CPU-bound work, use multiprocessing (separate processes, each with their own GIL) or C extensions (NumPy, Pandas) which release the GIL during heavy computation. Python 3.13+ has an experimental free-threaded mode (no GIL).

2. Reference Counting

CPython uses reference counting as its primary memory management strategy. Every object has a ob_refcnt field. When it drops to zero, the object is immediately deallocated — no GC pause needed for most objects.

Python — reference counting 
import sys

x = []               # refcount = 1 (x holds a reference)
y = x                # refcount = 2 (x and y both point to the list)
z = [x]              # refcount = 3 (x, y, and z[0] all point to it)

print(sys.getrefcount(x))   # 4 — getrefcount itself adds 1 temporarily

del y                # refcount drops to 3
del z                # refcount drops to 2 (z[0] also dropped)
del x                # refcount drops to 0 → list is immediately freed

# Small integer caching — CPython caches -5 to 256
a = 256
b = 256
print(a is b)   # True — same object (cached)

a = 257
b = 257
print(a is b)   # False — different objects (not cached)
                # This is why you should never use 'is' for equality checks

Weak References

Python 
import weakref

class Cache:
    pass

obj = Cache()
ref = weakref.ref(obj)      # doesn't increment refcount

print(ref())                # returns the object if alive
del obj
print(ref())                # None — object was collected

# WeakValueDictionary — values collected when no other references exist
cache = weakref.WeakValueDictionary()
task = Cache()
cache["key"] = task
print(cache.get("key"))     # Cache object
del task
print(cache.get("key"))     # None — automatically removed

3. Garbage Collection — Cyclic References

Reference counting fails for cycles — two objects that reference each other never reach refcount zero. CPython has a separate cyclic garbage collector for this.

Python 
import gc

# Create a reference cycle
class Node:
    def __init__(self, name: str):
        self.name = name
        self.next: "Node | None" = None

a = Node("A")
b = Node("B")
a.next = b          # a → b
b.next = a          # b → a  ← cycle!

# Delete external references
del a, del b
# Both still have refcount > 0 because of each other
# They will NOT be collected by refcounting alone

# Force cyclic GC
gc.collect()        # finds and collects cycles
# Now both are freed

# GC stats
print(gc.get_count())       # (gen0, gen1, gen2) object counts
print(gc.get_threshold())   # collection thresholds
gc.disable()                # turn off cyclic GC (if refcounting is enough for your app)

# Find what's keeping an object alive
import gc
class Leaking:
    pass

obj = Leaking()
print(gc.get_referrers(obj))   # lists all objects that reference obj
tracemalloc — the practical tool for finding memory leaks:
Python 
import tracemalloc

tracemalloc.start()

# ... run your code ...
data = [list(range(1000)) for _ in range(1000)]

snapshot = tracemalloc.take_snapshot()
top_stats = snapshot.statistics("lineno")

print("Top 5 memory allocations:")
for stat in top_stats[:5]:
    print(stat)

4. Object Identity vs Equality

This trips up Java developers because == means different things in the two languages.

Java 
String a = new String("hello");
String b = new String("hello");

a == b;          // false — reference identity
a.equals(b);     // true — value equality

Integer x = 127;
Integer y = 127;
x == y;          // true — cached
x = 128; y = 128;
x == y;          // false — not cached
Python 
a = "hello"
b = "hello"
a is b      # True — string interning (CPython optimization)
a == b      # True — value equality

a = ["x"]
b = ["x"]
a is b      # False — different list objects
a == b      # True — same contents

# id() returns the memory address (identity)
print(id(a), id(b))   # different numbers

# RULE: use == for value comparison, 'is' only for None/True/False
x = None
if x is None:    # correct idiom
    pass
if x == None:    # works but wrong — can be overridden by __eq__
    pass

5. Python Bytecode with dis

CPython compiles Python to an intermediate bytecode before executing it. The dis module lets you inspect this bytecode — useful for understanding what Python actually does and explaining performance differences.

Python 
import dis

def add(x, y):
    return x + y

dis.dis(add)
# Output:
#   2           0 RESUME                   0
#   3           2 LOAD_FAST                0 (x)
#               4 LOAD_FAST                1 (y)
#               6 BINARY_OP               0 (+)
#              10 RETURN_VALUE

# Disassemble a comprehension
dis.dis(compile("[x*2 for x in range(10)]", "", "eval"))

# Compare: is vs ==
def identity_check(x): return x is None
def equality_check(x): return x == None
dis.dis(identity_check)   # IS_OP — single bytecode instruction
dis.dis(equality_check)   # COMPARE_OP — potentially calls __eq__

Code Objects and Compilation

def factorial(n: int) -> int:

Quick Reference

GIL rule
threads for I/O, multiprocessing for CPU

GIL is released during I/O; CPU-bound threads don't benefit from multi-core

Reference count
sys.getrefcount(obj)

Returns current refcount (+1 for the getrefcount call itself)

Force GC
import gc; gc.collect()

Collects cyclic garbage; usually not needed in normal code

Find referrers
gc.get_referrers(obj)

Lists all objects that hold a reference to obj — debug memory leaks

Identity check
x is None / x is True / x is False

Use 'is' only for singletons; use == for everything else

Object address
id(obj)

Memory address in CPython; uniqueness only guaranteed while object is alive

Disassemble function
import dis; dis.dis(my_function)

Shows CPython bytecode instructions

Memory profiling
import tracemalloc; tracemalloc.start()

Track allocations by line; take_snapshot() to compare before/after

Weak reference
import weakref; ref = weakref.ref(obj)

Doesn't prevent collection; ref() returns None after obj is freed

Wrap-up

  • The GIL means Python threads don't achieve CPU parallelism. This is by design — it makes CPython's C internals simpler and thread-safe without fine-grained locking. Use multiprocessing for CPU work, threads for I/O.
  • Reference counting frees most objects immediately when they go out of scope — no GC pause. The cyclic garbage collector only runs for objects involved in reference cycles, which are much rarer.
  • is checks object identity (same memory address). == checks value equality (calls __eq__). Only use is for None, True, and False.
  • Integer caching (-5 to 256) and string interning are CPython optimizations. Don't rely on them — they're implementation details that can change. Never use is to compare integers or strings.
  • dis.dis(func) is surprisingly readable and useful for understanding why two implementations have different performance characteristics. If you're confused why something is slow, look at its bytecode.

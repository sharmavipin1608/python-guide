Concurrency: threading + multiprocessing

Python has three concurrency models. Today covers threads (good for I/O) and processes (good for CPU). The GIL is the key fact that determines which to pick.

The GIL — Python's Big Concurrency Caveat

CPython (the standard Python interpreter) has a Global Interpreter Lock (GIL) — a mutex that allows only one thread to execute Python bytecode at a time. This is Python's biggest difference from Java's threading model.

What the GIL means in practice:
— CPU-bound work (number crunching, string processing): threads do NOT run in parallel. Multiple threads will NOT use multiple CPU cores. Use multiprocessing instead.
— I/O-bound work (HTTP requests, file reads, DB queries): threads work great! While one thread waits for I/O, the GIL is released and other threads run.
— The GIL is released during I/O operations, time.sleep(), and calls into C extensions (NumPy, etc.).
Python — GIL demonstrated 
import threading
import time

def cpu_task(n: int) -> int:
    """CPU-bound: count to n."""
    return sum(range(n))

N = 50_000_000

# Sequential: 2 tasks back-to-back
start = time.perf_counter()
cpu_task(N)
cpu_task(N)
print(f"Sequential: {time.perf_counter() - start:.2f}s")

# Threaded: 2 threads running "simultaneously"
start = time.perf_counter()
t1 = threading.Thread(target=cpu_task, args=(N,))
t2 = threading.Thread(target=cpu_task, args=(N,))
t1.start(); t2.start()
t1.join();  t2.join()
print(f"Threaded:   {time.perf_counter() - start:.2f}s")
# Both take ~the same time! GIL prevents true parallelism for CPU work.

threading — I/O-Bound Concurrency

Basic threading.Thread

Java 
Thread t = new Thread(() -> {
    System.out.println("Running in thread: " +
        Thread.currentThread().getName());
});
t.start();
t.join(); // wait for completion
Python 
import threading
import time

def worker(name: str, delay: float) -> None:
    print(f"[{name}] starting")
    time.sleep(delay)   # simulates I/O wait
    print(f"[{name}] done after {delay}s")

# Create and start threads
t1 = threading.Thread(target=worker, args=("A", 1.0), name="thread-A")
t2 = threading.Thread(target=worker, args=("B", 0.5), name="thread-B")
t3 = threading.Thread(target=worker, args=("C", 0.8), name="thread-C")

start = time.perf_counter()
t1.start(); t2.start(); t3.start()

# join() blocks until that thread finishes
t1.join(); t2.join(); t3.join()

print(f"Total: {time.perf_counter() - start:.2f}s")
# ~1.0s — concurrent I/O! Sequential would be 2.3s

ThreadPoolExecutor — the right way

Managing individual threads is verbose. ThreadPoolExecutor is the idiomatic approach — it pools threads for reuse, returns Future objects, and handles exceptions cleanly. This is Python's ExecutorService.

Java 
ExecutorService pool =
    Executors.newFixedThreadPool(4);

List<Future<Integer>> futures = new ArrayList<>();
for (String url : urls) {
    futures.add(pool.submit(() -> fetch(url)));
}
for (Future<Integer> f : futures) {
    System.out.println(f.get());
}
pool.shutdown();
Python 
from concurrent.futures import ThreadPoolExecutor, as_completed
import requests

def fetch(url: str) -> tuple[str, int]:
    resp = requests.get(url, timeout=5)
    return url, len(resp.content)

urls = [
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
    "https://httpbin.org/delay/1",
]

# submit() — fire and forget, get Future back
with ThreadPoolExecutor(max_workers=4) as pool:
    futures = {pool.submit(fetch, url): url for url in urls}

    for future in as_completed(futures):
        url = futures[future]
        try:
            _, size = future.result()
            print(f"{url}: {size} bytes")
        except Exception as e:
            print(f"{url} failed: {e}")

# map() — simpler when you just need results in order
with ThreadPoolExecutor(max_workers=4) as pool:
    results = list(pool.map(fetch, urls))   # blocks until all done

Thread safety: Lock and Queue

Python 
import threading
from queue import Queue

# Lock — protect shared mutable state
counter = 0
lock = threading.Lock()

def increment(n: int) -> None:
    global counter
    for _ in range(n):
        with lock:         # acquire before, release after
            counter += 1  # critical section

threads = [threading.Thread(target=increment, args=(10_000,)) for _ in range(10)]
for t in threads: t.start()
for t in threads: t.join()
print(counter)   # always 100_000 — lock prevents race conditions


# Queue — thread-safe producer/consumer
q: Queue[str] = Queue()

def producer() -> None:
    for i in range(5):
        q.put(f"item-{i}")
        print(f"Produced item-{i}")
    q.put(None)   # sentinel to signal done

def consumer() -> None:
    while True:
        item = q.get()
        if item is None:
            break
        print(f"Consumed {item}")

t_prod = threading.Thread(target=producer)
t_cons = threading.Thread(target=consumer)
t_prod.start(); t_cons.start()
t_prod.join();  t_cons.join()

multiprocessing — CPU-Bound Parallelism

ProcessPoolExecutor — parallel CPU work

Each process gets its own Python interpreter and GIL — true multi-core parallelism. Processes don't share memory, so data is pickled (serialized) between them. Start-up cost is higher than threads.

Java — ForkJoinPool 
ForkJoinPool pool = new ForkJoinPool();
List<Future<Long>> futures = urls.stream()
    .map(n -> pool.submit(() -> heavyCompute(n)))
    .toList();
// Each task runs on a separate core
Python 
from concurrent.futures import ProcessPoolExecutor
import time

def cpu_heavy(n: int) -> int:
    """Simulate CPU-bound work."""
    return sum(i * i for i in range(n))

numbers = [5_000_000] * 4   # 4 heavy tasks

# Sequential
start = time.perf_counter()
results = [cpu_heavy(n) for n in numbers]
print(f"Sequential: {time.perf_counter() - start:.2f}s")

# Parallel — uses all CPU cores
start = time.perf_counter()
with ProcessPoolExecutor() as pool:   # defaults to os.cpu_count() workers
    results = list(pool.map(cpu_heavy, numbers))
print(f"Parallel:   {time.perf_counter() - start:.2f}s")
# On a 4-core machine: ~4x faster
Pickling requirement: Arguments and return values must be picklable (serializable). Lambda functions and locally-defined functions are NOT picklable — define your worker function at module level.

Decision Guide

ScenarioBest toolWhy
Many HTTP requestsThreadPoolExecutor or asyncioI/O-bound; GIL released during network wait
Read/write many filesThreadPoolExecutorI/O-bound; threads work well
Number crunching / data processingProcessPoolExecutor or NumPyCPU-bound; needs true parallelism
Image/video processingProcessPoolExecutorCPU-bound; each frame independent
Thousands of I/O operationsasyncio (Day 22)Threads have overhead; async scales better
Mix of I/O + CPUasyncio + ProcessPoolExecutorAsync for I/O, offload CPU to processes
Java developers note: In Java, threads parallelize CPU work because the JVM has no GIL. In Python, threads only help for I/O. This is the single most important concurrency fact to internalize.

Quick Reference Card

Raw thread
t = Thread(target=fn, args=(x,)); t.start(); t.join()

Low-level. Use ThreadPoolExecutor for most cases.

Thread pool
with ThreadPoolExecutor(max_workers=8) as p: p.map(fn, items)

Java's ExecutorService. Use for I/O-bound concurrent tasks.

Process pool
with ProcessPoolExecutor() as p: p.map(fn, items)

True CPU parallelism. Function must be defined at module level.

Future result
f = pool.submit(fn, arg); f.result(timeout=5)

result() blocks and raises any exception the task threw.

as_completed
for f in as_completed(futures): f.result()

Iterate results as they finish — don't wait for the slowest one first.

Lock
lock = Lock(); with lock: shared_state += 1

Protect shared mutable state. Use as context manager.

Thread-safe queue
q = Queue(); q.put(x); x = q.get()

The right way to pass data between threads. Thread-safe by design.

Daemon threads
t = Thread(target=fn, daemon=True)

Dies when main program exits. Good for background workers that shouldn't block shutdown.

Wrap-up

  • The GIL means Python threads cannot parallelize CPU-bound work — adding threads to a number-crunching loop makes it slower due to context-switch overhead, not faster. This is the opposite of Java's threading model.
  • Threads ARE useful for I/O-bound work (HTTP, DB, file I/O) because the GIL is released during I/O operations. A pool of 20 threads making HTTP requests concurrently is faster than doing them sequentially.
  • ThreadPoolExecutor and ProcessPoolExecutor share the same interface — both have submit(), map(), and work as context managers. Switching between them is often a one-line change.
  • Protect shared mutable state with threading.Lock as a context manager. Use queue.Queue for producer/consumer patterns — it's thread-safe by design, unlike a plain list.
  • For I/O-bound work at scale (hundreds or thousands of concurrent operations), asyncio (Day 22) is more efficient than threads — it handles all concurrency on a single thread with no GIL concerns and much less memory overhead.
